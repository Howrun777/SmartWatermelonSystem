# 基于ESP32的西瓜无损糖度智能检测系统设计与实现

---

## 摘要

西瓜是全球范围内广受欢迎的水果，其品质评价指标中糖度（°Brix）是决定口感与商业价值的最关键因素。传统糖度检测主要依赖手持折光仪进行有损取样测量，无法满足田间大规模普查和消费端现场快速无损检测的需求；专业近红外光谱分析仪器虽然精度高，但设备体积大、成本高昂，难以在基层推广应用。针对上述痛点，本文设计并实现了一套面向西瓜种植端与消费端的便携式智能糖度无损检测系统。

本系统以ESP32-WROOM-32双核微控制器为核心主控，采用AS7341多光谱传感器（11通道）进行西瓜表面光谱特征采集，基于多元线性回归（MLR）模型实现糖度的本地离线预测，检测结果以°Brix为单位实时显示于2.8寸TFT触控屏上。系统创新性地实现了"测试模式"与"工业模式"双模式业务架构：测试模式面向消费者或收购商，提供单次检测、累计均值计算和本地历史记录功能，无需网络连接即可独立工作；工业模式面向规模化种植采集场景，支持5分钟定时自动采集、LittleFS本地离线存储、WiFi恢复后随机延迟补发等完整数据链路功能，确保断网环境下数据零丢失。系统采用DS3231实时时钟（RTC）结合服务器时间戳双校时机制，保证采集时间戳的准确性。

后端数据服务层采用C++语言基于httplib库构建轻量级HTTP服务器，接入MySQL数据库进行西瓜糖度、成熟度评分和瓜田环境数据的持久化存储，并通过libsodium的Argon2id算法保障管理员账户安全。前端可视化平台采用原生HTML/CSS/JavaScript构建，集成ECharts图表引擎实现西瓜糖度历史曲线和温湿度/光照环境数据的实时可视化展示，支持多瓜田自动轮询切换。

经功能与性能测试验证，本系统糖度检测结果范围为0～20°Brix，支持最长72小时连续无故障运行，可稳定存储超过1000条离线数据记录，WiFi恢复后的断网补发成功率达100%。系统具备成本低、便携性强、功能完善、支持离线使用等优点，能够有效满足西瓜种植端田间品质监控和消费端现场快速检测的双场景需求，具备良好的实际应用价值与推广前景。

**关键词**：ESP32；近红外光谱；无损检测；糖度检测；AS7341多光谱传感器；嵌入式系统；离线数据存储；LittleFS

## Abstract

Watermelon is one of the most popular fruits worldwide, and its sugar content (°Brix) is the most critical factor determining taste and commercial value. Traditional sugar content detection mainly relies on handheld refractometers for destructive sampling, which cannot meet the needs of large-scale field surveys or consumer-side rapid non-destructive testing. Professional near-infrared spectral analysis instruments offer high accuracy but suffer from large size and high cost, making them difficult to promote at the grassroots level. To address these pain points, this thesis designs and implements a portable intelligent non-destructive sugar content detection system for watermelon, targeting both planting and consumer scenarios.

The system uses the ESP32-WROOM-32 dual-core microcontroller as the core controller, adopts the AS7341 multi-spectral sensor (11 channels) for watermelon surface spectral feature acquisition, and implements local offline sugar content prediction based on a Multiple Linear Regression (MLR) model, with detection results displayed in real-time on a 2.8-inch TFT touch screen. The system innovatively implements a dual-mode business architecture of "Test Mode" and "Industrial Mode": Test Mode targets consumers or buyers, providing single detection, cumulative average calculation, and local history recording without network connection; Industrial Mode targets large-scale planting and collection scenarios, supporting 5-minute timed automatic acquisition, LittleFS local offline storage, random delay data recovery after WiFi restoration, ensuring zero data loss in offline environments. The system adopts a DS3231 real-time clock (RTC) combined with server timestamp dual calibration mechanism to ensure accurate collection timestamps.

The backend data service layer uses C++ language based on the httplib library to build a lightweight HTTP server, connects to MySQL database for persistent storage of watermelon sugar content, maturity scores, and field environment data, and uses libsodium's Argon2id algorithm to ensure administrator account security. The frontend visualization platform uses native HTML/CSS/JavaScript to build, integrates the ECharts charting engine to achieve real-time visualization of watermelon sugar content history curves and temperature/humidity/light environment data, supporting automatic polling and field switching.

Through functional and performance testing, the system sugar content detection range is 0~20°Brix, supports up to 72 hours of continuous fault-free operation, can stably store more than 1000 offline data records, and achieves 100% success rate for data recovery after WiFi restoration. The system has the advantages of low cost, strong portability, complete functions, and offline support, which can effectively meet the dual-scenario needs of field quality monitoring at watermelon planting end and rapid on-site detection at consumer end, demonstrating good practical application value and promotion prospects.

**Keywords**: ESP32; Near-Infrared Spectroscopy; Non-Destructive Testing; Sugar Content Detection; AS7341 Multi-Spectral Sensor; Embedded System; Offline Data Storage; LittleFS

## 第1章 绪论

### 1.1 研究背景与意义

#### 1.1.1 西瓜产业现状与品质检测需求

西瓜（Citrullus lanatus）是葫芦科一年生蔓生草本植物，原产于非洲热带地区，现已在全球范围内广泛种植。中国是世界上最大的西瓜生产国和消费国，据国家统计局数据显示，中国西瓜年产量常年稳定在6000万吨以上，约占全球总产量的60%以上，西瓜产业在促进农村经济发展和满足居民消费需求方面发挥着重要作用。

在西瓜产业链中，糖度是决定西瓜口感品质和商业价值的核心指标。成熟的优质西瓜中心糖度通常在11°Brix～14°Brix之间，糖度低于10°Brix的西瓜通常口感偏淡，糖度达到12°Brix以上则被消费者普遍认为是"甜西瓜"。因此，糖度检测是西瓜品质分级、收购定价和田间采收决策的重要依据。

#### 1.1.2 传统检测方法的局限性

目前，西瓜糖度检测主要采用以下几种方法，各存在明显不足：

**折光仪法（有损检测）**：利用手持数字折光仪直接测量西瓜汁液的折射率来推算糖度，是目前最通用的检测方法。该方法测量精度较高（可达±0.1°Brix），但属于有损检测，需要切开后取汁取样，每次测量都意味着对西瓜的破坏，无法满足田间批量普查和消费端现场快速检测的需求。

**目测敲击法**：通过敲击西瓜听声音判断成熟度，完全依赖经验判断，准确性低，不同检测者之间结果差异大，无法给出量化的糖度数据，已逐渐被现代检测手段取代。

**专业近红外光谱分析法**：基于近红外光谱技术对西瓜内部成分进行无损分析，精度高，可同时测量糖度、酸度等多个指标。但专业光谱分析仪（如FOSS、Tec5等品牌）价格高昂（单台价格从数万元到数十万元不等），体积庞大需要专业操作人员，主要应用于大型农产品检测中心和研究机构，无法在田间地头或商超等基层场景普及。

#### 1.1.3 便携式近红外检测技术的应用价值

随着物联网技术和嵌入式系统的高速发展，低成本、高性能的可穿戴和便携式近红外检测设备成为果蔬品质检测领域的重要研究方向。将专业光谱检测技术小型化、嵌入式化，开发面向农业生产端和消费端的便携式糖度检测工具，具有以下重要价值：

**经济价值**：低成本设备可降低瓜农的检测设备采购门槛，帮助其在采收前进行批量糖度筛查，实现优质优价，提高经济效益。

**实用价值**：便携式设备可随身携带，支持田间地头、商超摊位等多种场景的现场快速检测，无需专业培训即可操作使用。

**社会价值**：推动西瓜等果蔬品质检测技术的普及应用，有助于提升农产品质量监管水平，保障消费者权益，促进农业信息化和数字化转型升级。

### 1.2 国内外研究现状

#### 1.2.1 国外研究现状

近红外光谱（Near-Infrared Spectroscopy, NIRS）技术在水果内部品质检测领域的应用研究始于20世纪80年代。国外学者在该领域的研究起步较早，已取得了丰硕成果。

在**检测原理与模型优化**方面，日本学者Kawano等（1992年）利用近红外透射光谱对西瓜糖度进行预测，建立了糖度预测偏最小二乘（PLS）回归模型，决定系数R²达到0.91。荷兰WUD集团的研究团队开发了商用便携式近红外水果无损检测仪VisionRT，可在1秒内同时测量糖度、酸度和果肉硬度，已在欧洲果蔬批发市场推广应用。

在**嵌入式便携设备研究**方面，近年来随着ESP32、树莓派等低成本高性能嵌入式平台的普及，涌现出大量面向农业场景的便携式检测设备研究。美国加州大学戴维斯分校的研究团队开发了基于树莓派和微型光谱传感器的便携式水果糖度检测装置，硬件成本控制在100美元以内，支持蓝牙数据传输和移动端APP显示。西班牙瓦伦西亚理工大学的研究者利用AS7341多光谱传感器对柑橘类水果进行糖度预测研究，验证了多光谱传感器在低功耗便携设备中用于果蔬品质检测的可行性。

在**系统集成与物联网应用**方面，澳大利亚联邦科学与工业研究组织（CSIRO）提出了"智慧果园"（Smart Orchard）概念，将便携式光谱检测终端与云平台数据服务结合，实现水果从种植到销售的全程品质追溯。

#### 1.2.2 国内研究现状

国内在近红外光谱果蔬品质检测领域的研究起步相对较晚，但近年来发展迅速，在理论研究和应用实践方面均取得了重要进展。

在**算法与模型研究**方面，中国农业大学、南京农业大学等高校在水果糖度近红外预测模型方面开展了系统研究。浙江大学的研究团队利用多种预处理方法（MSC、SNV、一阶导数等）结合PLS回归，显著提升了苹果糖度预测模型的稳定性和预测精度。中国农业科学院蔬菜花卉研究所针对西瓜糖度检测开展了专项研究，建立了西瓜糖度的特征波长筛选方法和最优PLS模型。

在**嵌入式检测设备研究**方面，浙江大学、西北农林科技大学等高校在便携式果蔬检测设备方面有较多研究积累。西北农林科技大学团队开发了基于STM32单片机的便携式番茄糖度检测仪，集成白光LED光源和硅光电池探测器，糖度预测误差约为±0.5°Brix。中国农业大学开发的基于ESP32和TCS34725颜色传感器的番茄成熟度检测装置，通过颜色特征参数与糖度的相关性分析实现了成熟度的快速判定。

在**物联网数据平台研究**方面，哈尔滨工业大学、南京农业大学等高校在农业物联网架构方面进行了深入研究，设计了面向规模化种植场景的传感网数据采集、传输和存储一体化解决方案。阿里云IoT、华为云IoT等国内主要云平台也推出了面向农业场景的物联网开发套件，支持快速接入多种传感器设备并实现数据上云。

#### 1.2.3 现有研究的不足

综合国内外研究现状，当前西瓜等果蔬糖度无损检测技术仍存在以下不足：

**功能单一性问题**：现有研究大多仅关注检测算法和精度，缺乏对设备业务功能的完整设计。许多设备仅支持单次检测，不具备连续采集、本地存储和离线工作能力，无法适应规模化种植场景的连续监测需求。

**离线能力缺失**：大多数便携式检测设备依赖实时网络连接，断网后即无法使用。在田间地头等网络条件不佳的场景中，这一问题严重制约了设备的实用性。

**双场景适配不足**：现有研究未能很好地区分"消费端快速检测"和"生产端批量采集"两种使用场景的差异化需求，往往采用统一设计而牺牲了某一场景的用户体验。

**成本与便携性矛盾**：专业设备精度高但成本高昂，低成本方案往往精度不足，缺乏在成本、精度、便携性三方面取得良好平衡的实用方案。

### 1.3 研究内容与技术路线

#### 1.3.1 主要研究内容

针对现有研究的不足，本文确定了以下主要研究内容：

**（1）硬件系统设计与集成**

设计并搭建以ESP32-WROOM-32为核心的硬件平台，集成AS7341多光谱传感器、SHT31温湿度传感器、BH1750光照传感器、DS3231实时时钟模块、2.8寸TFT触控显示屏等外设，实现传感器模块化选型与I2C总线集成设计，优化系统供电方案和硬件布局。

**（2）嵌入式软件架构设计与实现**

基于PlatformIO和Arduino-ESP32框架，采用驱动层→核心服务层→业务逻辑层→UI交互层的四层软件架构，设计并实现传感器驱动、WiFi网络通信、本地数据存储（LittleFS）、LVGL图形界面等核心软件模块。重点实现双模式业务逻辑（测试模式/工业模式）、双校时时间管理机制和离线数据补发策略。

**（3）后端数据服务系统开发**

基于C++语言和httplib轻量级HTTP框架，开发设备认证、糖度计算、成熟度评分、数据存储等后端服务模块。采用MySQL数据库存储西瓜糖度、成熟度和瓜田环境数据，通过libsodium的Argon2id算法保障管理员账户安全。提供符合RESTful规范的设备端和Web前端双套API接口。

**（4）前端可视化平台开发**

基于原生HTML/CSS/JavaScript和ECharts图表库，开发Web端瓜田监测可视化平台，实现西瓜糖度历史曲线和瓜田温湿度/光照环境数据的实时可视化展示，支持多瓜田自动轮询切换和图表时间尺度调节。

**（5）系统集成测试与验证**

搭建完整的测试环境，对系统的糖度检测精度、响应时间、连续运行稳定性、离线存储与补发能力等进行全面测试，验证系统的功能完备性和性能指标。

本系统的技术路线遵循"需求分析→总体设计→硬件实现→软件设计→后端搭建→系统集成→测试验证"的工程化流程，各阶段工作相互衔接、迭代推进，具体如下：

**第五阶段：前端平台开发**

**第六阶段：系统集成测试**

进行分模块测试和全流程集成测试，验证系统的各项功能和性能指标，分析测试结果并优化改进。

## 第2章 系统总体设计与相关技术基础

### 2.1 系统需求分析

#### 2.1.1 功能性需求分析

通过前期调研和需求梳理，本系统的功能性需求可归纳为以下几个主要方面：

**（1）时间同步需求**

系统需要在设备启动时获取准确的时间基准，用于为每次糖度检测打上可靠的时间戳。具体需求包括：设备启动后应自动连接WiFi并与服务器进行时间握手；服务器时间戳应优先于本地RTC时钟作为时间基准；当DS3231实时时钟模块存在时，断电期间应能保持时间连续性；网络断开时应能依赖RTC时钟维持计时功能。

**（2）光谱数据采集与糖度检测需求**

系统需要利用AS7341多光谱传感器采集西瓜表面的反射光谱数据，并通过内置算法计算糖度值。具体需求包括：光谱采集应覆盖至少8个可见光通道，采样结果以JSON格式存储和传输；糖度计算结果范围应覆盖西瓜实际糖度区间（0～20°Brix）；糖度值应实时显示在设备屏幕上；糖度算法应支持本地离线计算，不依赖网络。

**（3）环境参数采集需求**

为支撑瓜田环境监测功能，系统需要采集温度、湿度和光照强度等环境参数。具体需求包括：温度检测范围应覆盖田间实际温度区间（-40℃～+85℃），精度优于±0.5℃；湿度检测范围应覆盖0～100%RH，精度优于±3%RH；光照检测范围应覆盖0～65535 Lux；环境传感器应为可选模块，任一传感器缺失不应影响核心检测功能。

**（4）本地数据存储需求**

系统需要在无网络环境下暂存检测数据，并在网络恢复后将数据补发至服务器。具体需求包括：本地存储应采用非易失性存储器，确保断电后数据不丢失；存储容量应支持至少1000条历史记录；存储空间低于20%时应自动清理早期已上传数据；数据文件格式应便于追加读写。

**（5）双模式业务需求**

系统需要支持"测试模式"和"工业模式"两种工作模式，满足不同场景的需求。

测试模式（面向消费者/收购商）：支持单次糖度检测并在屏幕上显示结果；支持连续检测并计算累计均值（最多保存20条历史记录）；不依赖网络连接，检测数据不上传；支持清除历史数据操作。

工业模式（面向规模化种植采集）：支持手动触发和定时自动触发两种检测方式（自动触发间隔5分钟）；每次检测数据均上传至服务器；检测数据同步存储到本地；支持WiFi断线重连后自动补发未上传数据；支持查看历史检测记录及其上传状态。

**（6）人机交互需求**

系统需要通过TFT触控屏提供友好的人机交互界面。具体需求包括：设备启动时自动进入启动页显示设备信息和时间同步状态；时间同步完成后自动显示模式选择界面；测试模式和工业模式各有独立的操作界面；界面切换响应时间应小于100ms；界面设计应简洁直观，适合非专业人员操作。

**（7）后端数据服务需求**

后端系统需要为设备端和Web前端提供数据服务接口。具体需求包括：提供设备身份认证和Token校验机制；提供数据上传接口，完成糖度计算、成熟度计算和数据存储；提供时间同步接口；提供瓜田信息查询接口；提供管理员登录和Session会话管理；数据存储应持久化到MySQL数据库。

**（8）Web前端可视化需求**

Web端管理平台需要为管理员提供数据查看和监控功能。具体需求包括：支持多瓜田数据切换查看；实时显示各瓜田西瓜糖度和成熟度状态；支持糖度历史数据折线图展示；支持环境参数（温湿度/光照）历史折线图展示；支持4档时间尺度切换（5分钟/2小时/1天/1周）；支持自动轮询切换瓜田（默认6秒间隔）。

#### 2.1.2 非功能性需求分析

除功能性需求外，本系统的非功能性需求还包括以下方面：

**（1）性能指标**

糖度检测误差：≤±0.5°Brix（相对于标准折光仪测量值）；系统启动时间：≤5秒（含传感器初始化、WiFi连接、LVGL界面渲染）；检测响应时间：≤2秒（从触发测量到屏幕显示结果）；页面切换响应时间：≤100ms；网络请求响应时间：≤500ms（后端接口）；连续无故障运行时间：≥72小时；最大离线数据存储量：≥1000条。

**（2）可靠性需求**

系统应具备断网容错能力，断网期间应能正常进行糖度检测和数据存储；网络恢复后应能自动检测并启动数据补发流程；补发数据应保持原始时间戳的准确性；应防止重复数据入库。

**（3）安全性需求**

设备上传数据前应通过device_id和token双重校验；Web管理平台登录应使用强密码哈希（Argon2id）保护用户密码；Session会话应有有效期限制（默认1小时）；敏感配置信息（密码、token）不应明文暴露。

**（4）可维护性需求**

硬件模块采用标准化接口，支持快速更换；软件模块采用分层架构，便于后续功能扩展；后端接口遵循RESTful规范，便于第三方系统集成。

**（5）成本与功耗需求**

单台设备硬件成本应控制在300元以内；设备支持USB 5V供电；ESP32典型工作电流约80mA（240MHz双核），支持移动电源供电。

### 2.3 相关技术基础

#### 2.3.1 近红外光谱糖度检测原理

近红外光（Near-Infrared, NIR）是指波长范围在780nm～2526nm之间的电磁波，位于可见光红端与中红外光之间。近红外光谱区主要对应分子中C-H、O-H、N-H等化学键的伸缩振动和弯曲振动的倍频与合频吸收带，这些吸收特征与物质的化学成分和内部结构密切相关。

**朗伯-比尔定律（Lambert-Beer Law）**是近红外光谱定量分析的理论基础。当一束平行单色光通过均匀的吸收介质时，介质的吸光度（A）与吸光物质的浓度（c）和光程长度（l）成正比：

$$A = \log_{10}\frac{I_0}{I} = \varepsilon \cdot c \cdot l$$

其中，$I_0$为入射光强度，$I$为透射光强度，$\varepsilon$为摩尔吸光系数，$c$为吸光物质浓度，$l$为光程长度。

在西瓜糖度检测中，糖分（主要是蔗糖、葡萄糖和果糖）对近红外光有特定的吸收特征。当近红外光照射西瓜表面时，不同糖度的西瓜对不同波长光的反射率存在差异。通过检测多个波长点的反射光谱，并利用化学计量学方法建立光谱特征与糖度之间的定量关系模型，即可实现糖度的无损预测。

AS7341多光谱传感器覆盖了415nm～680nm的可见光范围以及白光（Clear）和近红外（NIR）通道，这些通道的波长分布覆盖了糖类物质的主要吸收区域，因此可以通过多元线性回归等方法建立光谱数据与糖度之间的关系模型。

**多元线性回归（MLR）**是一种常用的线性建模方法，假设目标变量（糖度）与多个自变量（光谱通道值）之间存在线性关系：

$$y = \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \cdots + \beta_n x_n + \varepsilon$$

其中，$y$为糖度预测值，$x_i$为各光谱通道的测量值，$\beta_i$为回归系数，$\beta_0$为截距，$\varepsilon$为随机误差。通过前期实验标定数据，利用最小二乘法估计各回归系数，即可得到可用于在线预测的糖度模型。

#### 2.3.2 嵌入式通信协议（I2C/SPI）

**I2C（Inter-Integrated Circuit）**是由Philips公司于1980年代开发的串行通信协议，仅需两根信号线（SDA数据线和SCL时钟线）即可实现多个设备之间的通信，具有接线简单、协议开销小的优点，在嵌入式系统中应用极为广泛。

I2C协议的基本特性包括：

总线结构：采用开漏输出加外部上拉电阻的结构，支持多个主设备和多个从设备共享同一总线。总线空闲时SDA和SCL均为高电平。

通信速率：标准模式为100kbit/s，快速模式为400kbit/s，快速模式+为1Mbit/s，高速模式为3.4Mbit/s。本系统采用标准模式100kbit/s，兼顾通信可靠性和距离。

寻址方式：每个从设备有唯一的7位或10位地址。主设备通过发送从设备地址+读/写位来选中目标设备进行通信。

通信流程：主设备发送起始条件（SCL高时SDA由高变低），然后发送从设备地址和读写位，被选中的从设备发送应答位（ACK），之后主设备和从设备之间进行数据传输，最后主设备发送停止条件（SCL高时SDA由低变高）结束通信。

本系统中的AS7341、SHT31、BH1750和DS3231均通过I2C总线与ESP32通信，它们的I2C地址分别为0x39、0x44、0x23和0x68，各设备地址互不冲突，可共用同一组I2C总线。

**SPI（Serial Peripheral Interface）**是由Motorola公司开发的同步串行通信协议，采用主从架构，通过四根信号线实现全双工通信：SCLK（时钟线）、MOSI（主机输出从机输入）、MISO（主机输入从机输出）和CS（片选线）。

SPI协议的基本特性包括：

通信速率：最高可达数十MHz，远高于I2C，适合大数据量传输。

数据宽度：可配置为4位～16位，本系统TFT显示屏采用16位数据宽度（RGB565格式）。

时钟极性和相位：SPI有4种时钟模式（CPOL和CPHA组合），本系统TFT显示屏采用Mode 0（SCLK空闲低电平，第一个时钟沿采样）。

本系统中的2.8寸TFT液晶显示屏（ILI9341驱动芯片）和XPT2046触摸控制芯片均通过SPI总线与ESP32通信。

#### 2.3.3 LittleFS嵌入式文件系统

LittleFS是一种专为嵌入式Flash存储器设计的轻量级文件系统，由ARM公司于2019年发布，旨在解决传统文件系统（如FAT）在嵌入式场景中的局限性。

LittleFS的核心设计特点包括：

掉电安全（Power-Loss Resilience）：LittleFS采用日志结构设计，每次写入操作都先写入新数据块，待新数据块写成功后才标记旧数据块为可删除状态。即使写入过程中发生突然断电，已写入的数据也不会损坏，文件系统可以在重启后自动恢复到一个一致的状态。

低RAM占用：LittleFS的RAM占用仅为几百字节到几KB，适用于RAM资源有限的微控制器。相比之下，FAT文件系统在缓存目录项时需要占用较大的RAM空间。

磨损均衡（Wear Leveling）：嵌入式Flash芯片的每个存储单元有固定的擦写寿命（通常为10万次），LittleFS通过动态管理数据块，将擦写操作分散到整个存储空间，延长Flash的使用寿命。

目录支持：LittleFS支持目录结构，便于数据的组织和管理。本系统中离线数据存储采用JSON Lines格式的单一文件，无需目录支持。

本系统使用ESP32内置的4MB Flash中的约2MB作为LittleFS分区（由PlatformIO自动分区），用于存储未上传的检测数据记录。

#### 2.3.4 HTTP网络通信与RESTful接口规范

HTTP（HyperText Transfer Protocol）是互联网上应用最为广泛的应用层协议，采用客户端-服务器架构，请求-响应模式工作。HTTP/1.1是目前最普遍使用的版本，支持持久连接（Keep-Alive）和管道化请求。

**HTTP请求方法**中，GET方法用于请求服务器发送指定资源，本系统设备端时间同步接口和Web端数据查询接口均使用GET方法。POST方法用于向服务器提交数据，本系统设备端数据上传接口和管理员登录接口使用POST方法，请求体采用JSON格式。

**HTTP请求头**中，Host指定目标服务器地址和端口；Content-Type指定请求体的MIME类型，本系统JSON接口使用application/json；自定义Header（如device_id、token）用于传递设备身份认证信息。

**HTTP响应状态码**中，200表示请求成功；400表示请求参数或数据格式错误；401表示认证失败；500表示服务器内部错误。

**RESTful（Representational State Transfer）**是一种设计Web API的软件架构风格，强调以资源为中心，通过HTTP方法表达对资源的操作。本系统的API设计遵循RESTful原则：

资源的URL路径使用名词复数形式，如/api/watermelon/list、/api/field/environment；HTTP方法表达操作语义，如GET查询资源、POST创建资源；响应格式统一使用JSON；无状态会话，认证信息通过请求头或Cookie传递。

#### 2.3.5 LVGL图形库

LVGL（Light and Versatile Graphics Library）是一款专为嵌入式系统设计的开源图形库，具有功能丰富、资源占用适中、易于移植等优点，被广泛应用于智能手表、工业HMI、消费电子等领域。

LVGL的核心特性包括：

丰富的图形组件：提供按钮、标签、列表、图表、滑动条、键盘等数十种GUI组件，支持分层视图（layers）和动画效果。

低资源占用：RAM占用约2KB～16KB（取决于配置），Flash占用约40KB～80KB，适合在ESP32等微控制器上运行。

跨平台支持：LVGL本身不依赖底层硬件，通过注册显示驱动和输入驱动实现跨平台移植，支持多种显示控制器和触摸芯片。

事件驱动架构：LVGL采用事件驱动编程模型，用户通过注册回调函数（Callback）来处理按钮点击、滑动等用户交互事件，代码结构清晰，易于维护。

本系统使用LVGL v8.3.9版本，配置为240×320分辨率、16位色深的显示驱动和电阻触摸输入驱动，实现了启动页、模式选择页、测试模式页和工业模式页共4个主要界面。

### 3.2 核心主控模块设计

#### 3.2.1 主控芯片选型

本系统选用乐鑫公司的ESP32-WROOM-32作为主控芯片。ESP32系列是当前物联网领域最受欢迎的无线微控制器之一，相比同级别的STM32F1、STM32F4和ESP8266等竞品，在无线功能、性能功耗比、生态支持等方面具有明显优势。

ESP32-WROOM-32的核心参数如下：

处理器架构：Xtensa LX6双核处理器，主频240MHz，支持单指令多数据（SIMD）指令扩展，DSP指令支持硬件加速，适合信号处理运算。

无线通信：集成WiFi（802.11 b/g/n）和蓝牙4.2/BLE，支持Station模式、SoftAP模式和混合模式，内置天线，无需额外无线模块。

存储资源：4MB SPI Flash，520KB SRAM，支持外接PSRAM扩展。

外设接口：丰富的数字外设，包括SPI（×4）、I2C（×2）、UART（×3）、I2S（×2）、CAN（×1）、IR（×1）、PWM（×16）、ADC（18通道）、DAC（×2）、电容触摸传感器（×10）等。

功耗特性：典型工作电流约80mA（240MHz双核），支持Light-Sleep和Deep-Sleep低功耗模式。

工作电压：3.0V～3.6V。

开发环境：支持Arduino-ESP32、ESP-IDF、PlatformIO等多种开发框架，生态成熟，文档完善，社区活跃。

选型对比分析中，STM32F4系列（如STM32F407）虽然处理器性能较强，但缺乏内置WiFi，需要额外外接ESP8266等无线模块，增加成本和布线复杂度。ESP8266虽然成本更低，但仅为单核处理器（80MHz），RAM仅50KB，外设接口数量和性能均不及ESP32。树莓派Pico（RP2040）为双核处理器，但同样缺乏内置WiFi，需要外接模块。因此，综合考虑功能完整性、性能功耗比、开发便利性和成本，ESP32-WROOM-32是本系统的最优选择。

ESP32 DevKit V1是乐鑫官方推出的基于ESP32-WROOM-32模组的开发板，板载USB转TTL芯片（CP2102或CH340），可直接通过microUSB线连接电脑进行程序烧录和串口调试，非常适合快速原型开发和学习研究。

#### 3.2.2 最小系统电路设计

ESP32 DevKit V1开发板已内置最小系统电路，包括电源电路、复位电路和下载电路。用户只需通过microUSB接口连接电脑即可完成供电和程序烧录，无需额外设计。

系统电源方案如下：USB接口提供5V电源，经过自恢复保险丝（500mA）和LC滤波电路后输入开发板LDO（低压差线性稳压器）转换为3.3V，为ESP32模组和传感器供电。传感器模块（有3.3V稳压芯片者）可共用USB 5V供电或3.3V供电。面包板电源模块设置为3.3V模式，为无独立稳压的传感器模块供电。

### 3.3 传感器采集模块设计

#### 3.3.1 光谱检测单元（AS7341）设计

AS7341是AMS公司推出的一款高性能多光谱传感器芯片，集成了8个可见光光谱通道、1个白光通道（Clear）和1个近红外通道（NIR），共11个独立光电二极管通道，能够获取物体表面丰富的光谱特征信息。

AS7341的核心参数如下：

通信接口：I2C，设备地址0x39，支持标准模式（100kHz）和快速模式（400kHz）。

光谱通道：共11个通道，各通道的中心波长和带宽分别为：ch415（415nm，紫色）、ch445（445nm，蓝色）、ch480（480nm，青蓝色）、ch515（515nm，绿色）、ch555（555nm，黄绿色）、ch595（595nm，黄色）、ch640（640nm，橙色）、ch680（680nm，红色）、Clear（白光，无滤光）、NIR（近红外，无滤光）。

ADC分辨率：16位，有效数值范围0～65535。

可编程增益：支持1×、2×、4×、8×、16×、32×、64×、128×、256×等多个增益等级，可根据光照条件自适应调整。

积分时间：可通过ATIME（积分周期数，范围0～255）和ASTEP（积分步长，0～65534）两个寄存器配置，积分时间 = ATIME × (ASTEP+1) × 2.78μs。

工作电压：3.3V。

数据格式：整数（0～65535）。

选型理由方面，AS7341相比传统的硅光电池+滤光片方案，无需设计复杂的光学结构，体积小巧，接口简单，非常适合便携式设备集成；相比专业光谱仪的低分辨率光谱仪（分辨率通常为5nm～10nm），AS7341虽然通道数有限，但已能捕捉糖分吸收区域的特征变化，结合MLR模型可实现糖度的实用级预测；相比同类多光谱传感器（如AS7262/AS7263），AS7341具有更高的动态范围和更多的通道数，测量精度更高。

AS7341的I2C接线设计：SDA引脚连接ESP32的GPIO27，SCL引脚连接ESP32的GPIO22，VCC连接3.3V，GND连接地线。设备地址固定为0x39，无需设置地址引脚。

驱动配置方面，本系统将ATIME配置为50（对应积分周期数），ASTEP配置为999（对应积分步长），增益（AGAIN）配置为256×。在此配置下，单次积分时间约为50×1000×2.78μs = 139ms，兼顾了测量速度和信噪比。

#### 3.3.2 环境参数采集单元（SHT31/BH1750）设计

**SHT31温湿度传感器**是Sensirion公司推出的高精度数字温湿度传感器，相比上一代SHT20在精度和稳定性方面有显著提升。

SHT31的核心参数如下：

通信接口：I2C，设备地址0x44（默认，可切换至0x45）。

温度测量范围：-40℃～+125℃。

温度测量精度：±0.2℃（典型值，25℃时），±0.5℃（全量程）。

湿度测量范围：0%～100%RH。

湿度测量精度：±2%RH（典型值），±5%RH（极端条件）。

工作电压：2.4V～5.5V（内置3.3V稳压器，可直接使用3.3V或5V供电）。

数据格式：浮点数，支持单次测量和周期测量模式。

SHT31的I2C接线设计：SDA引脚与AS7341共用的ESP32 GPIO27连接，SCL引脚与AS7341共用的ESP32 GPIO22连接，VCC连接3.3V，GND连接地线。由于SHT31内置了稳压器，可以安全地使用5V或3.3V供电。

**BH1750光照传感器**是ROHM公司推出的数字光照强度传感器，采用I2C接口输出直接量化的Lux值。

BH1750的核心参数如下：

通信接口：I2C，设备地址0x23（默认，可切换至0x5C）。

光照测量范围：0～65535 Lux。

光照测量精度：1 Lux。

工作电压：3.3V～5V。

功耗：约0.1mA（典型）。

测量模式：支持高分辨率模式（1Lux精度）、高分辨率模式2（0.5Lux精度）和低分辨率模式（4Lux精度）。

BH1750的I2C接线设计：SDA引脚与AS7341和SHT31共用的ESP32 GPIO27连接，SCL引脚与AS7341和SHT31共用的ESP32 GPIO22连接，VCC连接3.3V，GND连接地线。

**传感器解耦设计**是本系统的重要特性。系统设计为传感器"可选"模式：AS7341光谱传感器为必选（核心功能），SHT31和BH1750均为可选。当任一环境传感器缺失（I2C通信失败或未焊接）时，系统不报错也不影响核心功能，环境数据自动填99。这一设计大大增强了系统的容错能力和部署灵活性。

I2C总线多设备并联方面，AS7341（0x39）、SHT31（0x44）、BH1750（0x23）和DS3231（0x68）四个设备的SDA和SCL引脚并联到ESP32的GPIO27和GPIO22上，共用上拉电阻（4.7kΩ）。由于各设备I2C地址均不同，主设备（ESP32）通过地址访问实现对各设备的独立通信，不会产生地址冲突。

#### 3.3.3 实时时钟单元（DS3231）设计

DS3231是Maxim（现ADI）公司推出的高精度I2C实时时钟芯片，集成了温度补偿晶体振荡器（TCXO），在-40℃～+85℃全温度范围内的计时精度可达±2分钟/年，远优于普通晶振+软件计时方案。

DS3231的核心参数如下：

通信接口：I2C，设备地址0x68。

计时精度：±2ppm（0℃～+40℃），±3.5ppm（-40℃～+85℃）。

备用电源：支持外接CR2032纽扣电池，断电后可持续计时约1年。

时间寄存器：秒、分、时、星期、日、月、年（支持世纪进位），采用BCD编码。

闹钟功能：支持两个可编程闹钟，可用于定时唤醒ESP32。

工作电压：2.3V～5.5V。

DS3231的I2C接线设计：SDA引脚与AS7341、SHT31、BH1750共用的ESP32 GPIO27连接，SCL引脚与AS7341、SHT31、BH1750共用的ESP32 GPIO22连接，VCC连接3.3V，GND连接地线，VBAT引脚连接CR2032纽扣电池（正极接VBAT，负极接地）。

断电时间保持机制：当系统正常供电时，DS3231使用主电源供电，同时为备用电池充电；当主电源断开时，DS3231自动切换到VBAT引脚供电，由纽扣电池维持RTC时钟继续运行。这一机制确保了设备断电期间时间戳的连续性，再次上电后无需重新校时。

双校时逻辑方面，设备上电后的时间同步策略如下：首先尝试从DS3231读取当前时间作为初始时间；若DS3231不存在或通信失败，则使用ESP32内部软件时钟；WiFi连接成功后，主动调用时间同步接口获取服务器时间戳，并用服务器时间更新DS3231和ESP32内部时钟；后续每次数据上传后，从服务器响应中提取server_time字段，自动校准本地时钟。

### 3.4 人机交互模块设计

#### 3.4.1 TFT显示单元设计

本系统选用2.8寸TFT液晶显示屏，分辨率240×320像素，采用ILI9341驱动芯片，通过SPI接口与ESP32通信。

显示屏的核心参数如下：

驱动芯片：ILI9341，支持262K色（RGB565格式）。

分辨率：240×320像素。

通信接口：SPI，支持最高数十MHz时钟频率。

工作电压：3.3V。

显示方向：支持0°、90°、180°、270°四个方向旋转，本系统配置为0°。

ILI9341的SPI接线设计如下：SCLK（CLK）连接ESP32 GPIO18，MOSI连接ESP32 GPIO23（发送数据），MISO连接ESP32 GPIO19（读取触控数据），CS连接ESP32 GPIO5，DC（RS）连接ESP32 GPIO21，RST连接ESP32 GPIO22，背光控制BL连接ESP32 GPIO4，VCC连接3.3V，GND连接地线。

显示缓冲区配置方面，本系统为LVGL配置了双缓冲区（Dual Buffer），缓冲区大小为screenWidth × 10 = 240 × 10 = 2400像素，对应约4.8KB RAM（2400像素 × 2字节/像素）。双缓冲模式下，LVGL在一个缓冲区中渲染图像，同时通过DMA将另一个缓冲区内容发送到显示屏，渲染完成后交换缓冲区，实现流畅的显示效果。

#### 3.4.2 触摸交互单元设计

显示屏集成XPT2046电阻式触摸控制芯片，通过独立的SPI总线与ESP32通信。

XPT2046的核心参数如下：

通信接口：SPI，设备地址通过SPI片选控制（无需I2C地址）。

触控分辨率：12位（4096级），可识别0.01mm级别的位置变化。

支持触控点数：单点触控。

工作电压：3.3V。

XPT2046的SPI接线设计：利用TFT显示屏的MISO引脚作为XPT2046的数据输出，将TFT的CS引脚与XPT2046的CS引脚并联（共用的片选信号，由TFT_eSPI库自动控制）。由于TFT和XPT2046使用同一组SPI总线但片选信号不同，ESP32可以分时访问两个设备。具体接线为：SCLK连接ESP32 GPIO18（共用），MOSI连接ESP32 GPIO23（共用），MISO连接ESP32 GPIO39（XPT2046数据输出），CS连接ESP32 GPIO33（XPT2046片选），IRQ连接ESP32 GPIO36（触摸中断通知），VCC连接3.3V，GND连接地线。

触摸坐标映射方面，XPT2046返回的原始ADC值范围为0～4095，需要经过坐标映射转换到屏幕像素坐标（0～239，0～319）。本系统配置触摸旋转角度为0，通过`map()`函数将X轴原始值（250～3800）映射到0～239，将Y轴原始值（250～3800）映射到0～319。

LVGL集成方面，本系统配置LVGL的输入设备驱动类型为`LV_INDEV_TYPE_POINTER`（触摸指针设备），读取回调函数读取XPT2046的触摸状态和坐标，将触摸事件上报给LVGL的事件系统。LVGL内部自动处理按钮点击、滑动等手势识别。

### 3.5 光源模块设计

为消除环境光对光谱测量的干扰，本系统设计了3W宽谱白光LED灯带作为测量光源，在每次光谱采集前点亮，采集完成后关闭。

LED驱动电路设计方面，由于ESP32 GPIO的输出电流能力有限（单个GPIO最大12mA），而3W LED的工作电流约700mA（3W/5V≈600mA），GPIO无法直接驱动LED。本系统采用NPN三极管（S8050）作为开关管设计驱动电路：GPIO26通过330Ω限流电阻连接到三极管基极，三极管集电极连接LED灯带正极，LED灯带负极接地，发射极接地。当GPIO26输出高电平时，三极管导通，LED点亮；当GPIO26输出低电平时，三极管截止，LED熄灭。

LED控制时序方面，光谱采集流程为：启动LED → 等待500ms（LED稳定） → 读取AS7341光谱数据 → 关闭LED。等待500ms的稳定时间是为了消除LED点亮瞬间的瞬态响应，确保光谱数据的准确性。

#### 4.1.2 软件分层架构设计

本系统的嵌入式软件采用四层分层架构，从下至上依次为：驱动层（Driver Layer）、核心服务层（Core Service Layer）、业务逻辑层（Business Logic Layer）和UI交互层（UI Interaction Layer）。各层职责明确，层间通过定义良好的接口通信，体现了良好的软件工程设计思想。

**驱动层**位于软件架构的最底层，负责直接与硬件外设交互，封装硬件的寄存器操作和数据传输细节，为上层提供统一的硬件抽象接口。主要包括：SensorManager（传感器统一管理）、RTCManager（DS3231实时时钟驱动）、HttpClient（WiFi和HTTP通信）、StorageManager（LittleFS文件系统）、LightControl（LED光源驱动）。

**核心服务层**位于驱动层之上，封装了系统运行所需的基础服务功能，为业务逻辑层提供支撑。主要包括：时间同步服务（整合RTC和网络校时）、数据存储服务（封装StorageManager的操作逻辑）、网络通信服务（封装HttpClient的请求逻辑）、存储空间管理服务（监测容量、自动清理）。

**业务逻辑层**位于核心服务层之上，实现了系统的核心业务功能，是整个软件系统的"大脑"。主要包括：糖度计算算法（基于光谱数据的MLR模型）、成熟度计算算法（基于糖度和品种阈值）、双模式业务调度器（管理MODE_TEST和MODE_INDUSTRIAL两种模式的切换和数据流向）、测量引擎（协调传感器采集→糖度计算→数据存储→网络上传的完整流程）。

**UI交互层**位于软件架构的最顶层，负责用户界面的渲染和用户输入的处理，直接调用业务逻辑层完成用户请求。主要包括：启动页面（显示设备信息和时间同步状态）、模式选择页面（测试模式/工业模式入口按钮）、测试模式页面（糖度显示、历史表格、均值计算）、工业模式页面（实时环境数据、光谱数据、自动倒计时、历史表格）。

软件分层架构的优势体现在：模块独立性（各层可独立开发、测试和调试）、可复用性（驱动层和核心服务层可在其他项目中复用）、可维护性（问题定位和修复可快速定位到特定层次）、可扩展性（新增业务模式只需在业务逻辑层添加，无需修改底层代码）。

### 4.2 底层驱动层设计与实现

#### 4.2.1 传感器驱动开发

**SensorManager**是传感器驱动的核心类，采用单例模式（Singleton）设计，通过`getInstance()`获取唯一实例，统一管理AS7341、SHT31、BH1750三个I2C传感器的初始化、配置和数据读取。

SensorManager的初始化流程（`begin()`方法）如下：

```cpp
void SensorManager::begin() {
    Wire.begin(I2C_SDA_PIN, I2C_SCL_PIN);  // 初始化I2C总线
    Wire.setClock(100000);                  // 设置I2C时钟100kHz
    
    // 检测并初始化AS7341（必选）
    if (detect_AS7341()) {
        as7341.begin();
        configure_AS7341();  // 配置积分时间和增益
    }
    
    // 检测并初始化SHT31（可选）
    if (detect_SHT31()) {
        sht31.begin(0x44);
    }
    
    // 检测并初始化BH1750（可选）
    if (detect_BH1750()) {
        bh1750.begin();
        bh1750.configure(BH1750_CONTINUOUS_HIGH_RES_MODE);
    }
}
```

传感器检测采用"尝试通信→判断结果"的方式：向传感器的I2C地址发送写请求并等待应答，如果应答成功则认为传感器存在，如果超时无应答（返回0字节）则认为传感器不存在或通信失败。传感器检测失败不影响程序继续运行，体现了系统的鲁棒性设计。

光谱数据读取流程（`readSpectrum()`方法）如下：

```cpp

    if (!is_AS7341_Detected) return false;

    uint16_t ch415, ch445, ch480, ch515, ch555, ch595, ch640, ch680, clear, nir;
    as7341.readChannels(ch415, ch445, ch480, ch515, ch555, ch595, ch640, ch680, clear, nir);

    specObj["ch445"] = ch445;
    specObj["ch480"] = ch480;
    specObj["ch515"] = ch515;
    specObj["ch555"] = ch555;
    specObj["ch595"] = ch595;
    specObj["ch640"] = ch640;
    specObj["ch680"] = ch680;

```

环境数据读取流程（`readEnvironment()`方法）如下：尝试读取SHT31温湿度数据，如果传感器不存在或读取失败，温度和湿度返回值均为99；尝试读取BH1750光照数据，如果传感器不存在或读取失败，光照返回值也为99。这一设计确保了环境传感器的可选性，任一传感器缺失均不影响系统核心功能。

### 4.3 核心服务层设计与实现

#### 4.3.1 自适应时间同步管理模块

RTCManager采用单例模式设计，统一管理DS3231实时时钟和ESP32内部软件时钟，提供`getTimestamp()`、`syncTime()`、`formatTime()`等关键接口。

DS3231初始化（`begin()`方法）首先尝试I2C地址0x68通信，如果应答成功则表示DS3231存在，随后读取其时间寄存器获取当前时间。如果DS3231不存在（如未焊接模块），则系统仅依赖WiFi连接后的网络校时。

时间同步策略采用DS3231优先、网络校准的双层机制：

**第一层：DS3231硬件RTC**。设备上电后首先读取DS3231的时间寄存器，由于DS3231在断电期间由纽扣电池维持计时，上电时读取的时间即为断电前的真实时间。如果设备断电时间较短（<1年），RTC时间通常是准确的。

**第二层：网络服务器校准**。WiFi连接成功后，设备立即调用`/api/device/time`接口获取服务器时间戳。服务器返回的时间戳是UTC时间（Unix纪元以来的秒数），设备收到后通过`settimeofday()`系统调用更新ESP32的内部时钟，同时通过I2C写入DS3231的时间寄存器实现持久化。后续每次数据上传后，设备从服务器响应的`server_time`字段中提取最新时间，自动进行微调校准。

```cpp
// WiFi连接成功后的首次校时
if (NetworkManager::connectWiFi()) {
    uint32_t serverTime = NetworkManager::fetchServerTime();
    if (serverTime > 0) {
        RTCManager::getInstance().syncTime(serverTime);
    }
}

// 数据上传后的自动校准
if (uploadOk && serverTime > 0) {
    RTCManager::getInstance().syncTime(serverTime);
}
```

北京时间显示方面，DS3231和服务器返回的时间戳均为UTC标准时间，而中国标准时间（CST）为UTC+8。系统在`formatTime()`方法中对UTC时间戳统一加上8小时偏移量（28800秒），确保屏幕显示的时间为准确的北京时间。

#### 4.3.2 本地数据存储管理模块

StorageManager采用单例模式设计，基于LittleFS文件系统实现离线数据的持久化存储。

**存储格式设计**：本系统采用JSON Lines格式存储离线数据。JSON Lines是一种适合日志和数据流的文本格式，每行是一个独立的完整JSON对象，以换行符分隔，便于追加写入和逐行读取。存储文件位于`/data/watermelon_log.jsonl`。

一条离线数据记录的JSON格式如下：

```json
{"ts":1712345678,"brix":12.5,"t":25.3,"h":60.2,"l":10000,"spec":{"ch415":123,"ch445":456},"up":0}
```

其中，`ts`为数据采集时间戳，`brix`为糖度值，`t/h/l`为环境数据，`spec`为光谱数据，`up`为上传状态标记（0=未上传，1=已上传）。

**存储空间管理**：系统在每次保存新记录前调用`checkAndCleanCapacity()`检查存储空间使用率。当LittleFS已用空间超过80%（即剩余空间低于20%）时，自动删除最早的一半历史数据（包括已上传和未上传的），以腾出空间存储新数据。

```cpp
void StorageManager::checkAndCleanCapacity() {
    size_t total = LittleFS.totalBytes();
    size_t used = LittleFS.usedBytes();
    
    if ((total - used) < total * 0.2) {  // 剩余空间不足20%
        // 删除最早50%的历史记录
        cleanOldestHalfRecords();
    }
}
```

**上传状态标记**：当数据成功上传至服务器后，调用`markAsUploaded(target_ts)`方法，将指定时间戳对应记录的`up`字段从0更新为1：

```cpp
void StorageManager::markAsUploaded(uint32_t target_ts) {
    // 读取所有记录 → 修改目标记录的up字段 → 全部重写
}
```

**未上传数据查询**：调用`getUnuploadedRecord(recordObj, target_ts)`按时间顺序（从最早到最新）查找第一条`up=0`的记录，用于WiFi恢复后的离线数据补发。

#### 4.3.3 网络通信与数据补发模块

HttpClient封装了ESP32的WiFi连接和HTTP通信功能，提供`connectWiFi()`、`fetchServerTime()`和`uploadData()`三个核心接口。

**WiFi连接管理**（`connectWiFi()`方法）：

```cpp
bool NetworkManager::connectWiFi() {
    WiFi.mode(WIFI_STA);
    WiFi.begin(WIFI_SSID, WIFI_PASS);
    
    for (int i = 0; i < 20; i++) {
        if (WiFi.status() == WL_CONNECTED) return true;
        delay(500);
    }
    return false;  // 连接超时
}
```

最长等待时间为20次×500ms = 10秒。如果10秒内未能连接成功，则返回false，设备进入离线模式。

**WiFi后台维护**（`maintainWiFi()`方法）：在主循环的每次迭代中调用，检查WiFi连接状态。如果当前未连接且距离上次连接尝试已超过30秒，则重新尝试连接WiFi。

**数据上传**（`uploadData()`方法）：将糖度检测数据封装为JSON，发送HTTP POST请求到服务器，解析服务器响应并返回上传结果和server_time。

```cpp

    http.addHeader("Content-Type", "application/json");

        String response = http.getString();
        // 解析JSON响应，提取server_time
        // ...
        return true;

    }

```

**离线数据补发策略**：当WiFi从断开状态恢复时，系统启动数据补发状态机。首先随机等待0～10秒（`random(0, 10000)`），等待时间到后开始按序查找LittleFS中`up=0`的记录，逐一上传。每条记录上传成功后更新本地存储的`up`标记。如果WiFi再次断开，下次恢复时继续从中断处补发。

随机等待的设计目的是避免大量设备同时恢复WiFi时产生的"补发风暴"，防止服务器瞬时压力过大。实际部署时可将延迟上限调整为200秒（`random(0, 200000)`）以更好地错峰。

### 4.4 业务逻辑层设计与实现

#### 4.4.1 双模式业务逻辑设计

系统定义了`SystemMode`枚举类型表示两种工作模式：

```cpp
enum SystemMode {
    MODE_WAITING = 0,    // 等待时间同步状态（开机后的过渡状态）
    MODE_TEST = 1,       // 测试模式（消费者/收购商使用）
    MODE_INDUSTRIAL = 2  // 工业模式（规模化种植采集使用）
};
```

设备开机后首先进入MODE_WAITING状态，完成WiFi连接和时间同步后自动切换到模式选择界面，用户可选择进入MODE_TEST或MODE_INDUSTRIAL。模式选择按钮的显示时机由`showModeSelection()`方法控制——仅当时间同步完成后才显示，否则仅显示"Waiting for Time Sync..."状态。

**测试模式（Consumer Test Mode）的业务逻辑**如下：

测试模式面向消费者或收购商，提供快速、无网络依赖的糖度检测功能。每次点击"MEASURE"按钮后，系统执行以下流程：

```cpp
if (sysMode == MODE_TEST) {

    sensorMgr.controlLight(true);
    delay(500);
    
    // 2. 读取光谱数据
    JsonObject specObj;
    bool specOk = sensorMgr.readSpectrum(specObj);
    sensorMgr.controlLight(false);
    
    // 3. 环境数据不采集（测试模式不解耦环境传感器）
    float t = 99.0, h = 99.0; int l = 99;
    
    // 4. 本地计算糖度
    float brix = SugarCalc::calculate(specObj);
    
    // 5. 更新统计（内存，非持久化）
    if (test_count >= MAX_TEST_COUNT) onClearTestClicked();  // 超20条自动清空
    test_count++;
    test_sugar_sum += brix;
    float avg = test_sugar_sum / test_count;
    
    // 6. 更新UI
    ui.updateTestStats(brix, test_count, avg);
    ui.addTestHistory(test_count, brix, avg);
}
```

#### 4.4.2 糖度预测算法实现

本系统的糖度预测采用多元线性回归（MLR）模型。MLR是最基础的化学计量学建模方法，假设糖度值与各光谱通道的测量值之间存在线性组合关系：

$$\text{Sugar} = \beta_0 + \beta_1 \cdot ch_{415} + \beta_2 \cdot ch_{445} + \beta_3 \cdot ch_{480}$$

其中，$\beta_0$为截距，$\beta_1$、$\beta_2$、$\beta_3$为各通道的回归系数。

考虑到西瓜糖度的主要吸收特征波长集中在400nm～500nm范围（紫色到蓝色光区），以及糖类物质在480nm附近的吸收峰，本系统选择ch415、ch445和ch480三个通道构建MLR模型。

回归系数的标定需要前期实验数据的支撑。系数$a=0.0012$、$b=0.0005$、$c=0.0021$和截距$k=5.2$ 为初步估算值，可通过以下实验方法进行标定：

**标定实验步骤**：选取若干（建议不少于30个）已知糖度的西瓜样本，使用本系统设备测量其AS7341光谱数据，同时使用标准折光仪测量其真实糖度值；利用最小二乘法对光谱数据和糖度数据进行回归分析，得到最优的回归系数；将标定好的系数更新到`sugar_calc.h`中。

```cpp
// 糖度计算算法（sugar_calc.h）
namespace SugarCalc {
    double calculate(const nlohmann::json& spectrum) {
        // 多元线性回归模型
        double k = 5.2;    // 截距
        double a = 0.0012; // ch415系数
        double b = 0.0005; // ch445系数
        double c = 0.0021; // ch480系数
        
        double ch415 = spectrum.value("ch415", 0);
        double ch445 = spectrum.value("ch445", 0);
        double ch480 = spectrum.value("ch480", 0);
        
        double brix = a * ch415 + b * ch445 + c * ch480 + k;
        
        // 限制在合理范围内
        if (brix < 0)   return 0.0;
        if (brix > 20)  return 20.0;
        
        return brix;
    }
}
```

糖度值的合理性约束：西瓜的实际糖度范围通常在6°Brix～16°Brix之间，极端情况不会低于0°Brix或超过20°Brix。因此，算法对计算结果进行了上下限约束，确保输出值在合理范围内。

## 第5章 后端服务系统设计与实现

### 5.1 后端系统开发环境与总体架构

#### 5.1.1 开发环境与工具链

后端数据处理中心采用C++语言开发，开发环境配置如下：

**编译器与构建工具**：GCC/G++ 7.0以上版本（支持C++17标准），CMake 3.10以上版本作为构建工具。CMakeLists.txt配置文件定义了源文件组织、第三方库链接和编译选项。

**HTTP框架**：cpp-httplib（由@yhirose开发的轻量级C++ HTTP服务器库），仅需引入单个头文件即可使用，无需复杂的依赖安装，支持GET/POST请求处理、JSON响应、静态文件托管等常用功能。

**JSON处理**：nlohmann/json（由nlohmann开发的header-only JSON库），提供了与STL容器类似的JSON操作接口，支持JSON序列化/反序列化、路径查询、修改等功能。

**密码哈希**：libsodium（由Frank Denis开发的现代化密码学库），提供了Argon2id、XSalsa20、ChaCha20等安全算法。其中Argon2id是2015年Password Hashing Competition的获胜算法，是目前最先进的密码哈希算法之一，抗GPU/ASIC攻击能力强。

**数据库**：MySQL 8.0，使用C API（mysql-client library）进行数据库操作。

#### 5.1.2 后端系统架构

后端系统采用"控制层→服务层→数据访问层"的三层架构设计：

**控制层（Controller）**：由HttpServer和Router两个模块组成。HttpServer封装了httplib Server实例，负责监听端口（默认8080）和分发HTTP请求。Router模块集中注册所有路由，将不同的URL路径和HTTP方法映射到对应的处理函数。

**服务层（Service）**：包含DeviceAuth（设备认证）、AdminAuth（管理员认证）、SugarCalc（糖度计算）、MaturityCalc（成熟度计算）、DataCheck（数据校验）、Session（会话管理）等核心服务类。各服务类封装了独立的业务逻辑，Router处理函数调用相应的服务类完成业务处理。

**数据访问层（Data Access）**：由MySQLDriver和PasswordHasher组成。MySQLDriver封装了MySQL C API的连接、查询、结果集处理等操作。PasswordHasher封装了libsodium的密码哈希和验证功能。

后端系统的启动流程如下：

```cpp
int main() {
    // 1. 初始化 libsodium
    if (sodium_init() < 0) {
        std::cerr << "libsodium initialization failed" << std::endl;
        return 1;
    }
    
    // 2. 连接MySQL数据库
    bool db_ok = MySQLDriver::getInstance().connect(
        "127.0.0.1", "sws_user", "Mhr289839.", "sws_db", 3306);
    if (!db_ok) return 1;
    
    // 3. 自动迁移admin密码到Argon2格式
    std::string new_hash = PasswordHasher::hash("admin123");
    MySQLDriver::getInstance().execute(
        "UPDATE admin_users SET password_hash = '" + new_hash + "' WHERE username = 'admin'");
    
    // 4. 启动HTTP服务器
    HttpServer server;
    server.start("0.0.0.0", 8080);
    
    return 0;
}
```

设备端将糖度检测数据上传至服务器，服务端完成数据校验、糖度计算、成熟度计算和数据存储。

**请求头**：

| 参数       | 类型   | 必填 | 说明          |
| --------- | ------ | ---- | ----------- |
| device_id | String | 是   | 设备编号        |
| token     | String | 是   | 设备认证Token     |
| Content-Type | String | 是 | application/json |

**请求体**（JSON）：

```json
{
  "spectrum_json": {
    "ch415": 123, "ch445": 456, "ch480": 789,
    "ch515": 234, "ch555": 567, "ch595": 890,
    "ch640": 321, "ch680": 654, "chClear": 999, "chNIR": 111
  },
  "temperature": 25.5,
  "humidity": 60.2,
  "light": 10000,
  "collected_at": 1712345678
}
```

其中`spectrum_json`为必填项，`collected_at`为可选（不填则使用服务器当前时间）。

**响应示例**（HTTP 200）：

```json
{
  "code": 200,
  "msg": "数据上传成功",
  "data": {
    "sugar_brix": 12.5,
    "maturity_score": 1.02,
    "collected_at": 1712345678,
    "server_time": 1712345679
  }
}
```

**服务端业务处理流程**如下：

步骤1：设备身份认证。从HTTP请求头中提取`device_id`和`token`，调用`DeviceAuth::authenticate()`校验。认证失败返回HTTP 401，响应体为`{"code":401,"msg":"设备认证失败"}`。

步骤2：解析请求数据。从JSON请求体中提取`spectrum_json`（必填）、`temperature`、`humidity`、`light`和`collected_at`（可选）。

步骤3：获取采集时间戳。优先使用请求体中的`collected_at`字段值。如果请求体中无此字段，则使用服务器当前的Unix时间戳。

步骤4：提取瓜田号。从`device_id`中提取前缀（前4位，即`device_id.substring(0, 4)`，如`1001-01-01`→`1001`），作为`field_id`。

步骤5：数据合法性校验。调用`DataCheck::isFieldExist(field_id)`检查瓜田号是否存在于`field_production`表中，不存在则返回HTTP 400。

步骤6：糖度计算。调用`SugarCalc::calculate(spectrum_json)`，输入AS7341光谱数据，输出糖度值（°Brix）。计算公式为：

```cpp
double brix = 0.0012 * ch415 + 0.0005 * ch445 + 0.0021 * ch480 + 5.2;
if (brix < 0) brix = 0.0;
if (brix > 20) brix = 20.0;
```

步骤7：成熟度计算。调用`MaturityCalc::calculate(field_id, sugar_brix)`，从`field_production`表中查询该瓜田的`mature_sugar_threshold`，计算`maturity_score = sugar_brix / threshold`。

步骤8：西瓜数据入库。构造INSERT语句写入`watermelon_data`表。如果遇到联合主键冲突（同一设备同一秒已存在记录），则将`collected_at`加1秒后重试（最多重试10次）。

步骤9：环境数据入库。调用`DataCheck::isEnvironmentValid(temperature, humidity, light)`校验环境数据有效性。有效性规则为：不能temperature=99 **且** humidity=99 **且** light=99。如果有效，则将环境数据写入`field_environment`表（同样处理主键冲突）。

步骤10：返回响应。返回HTTP 200，响应体包含`sugar_brix`、`maturity_score`、`collected_at`和`server_time`字段。`server_time`字段供设备端下次校时使用。

#### 5.3.3 接口身份校验机制

本系统设计了设备端和Web前端两套身份校验机制：

**设备端Token认证**：设备上传数据时，在HTTP请求头中携带`device_id`和`token`两个字段。服务端在`Router`层对所有`/api/device/*`路由进行统一校验，调用`DeviceAuth::authenticate()`方法。该方法首先根据`device_id`查询`device_auth`表，获取该设备注册时的Token和状态，然后使用常量时间比较（`sodium_memcmp`）对比Token是否匹配，最后检查`status`字段是否为1（启用状态）。三步校验全部通过后才允许处理后续业务逻辑。

**Web管理员Session认证**：管理员登录后，服务端生成一个32字符的随机UUID作为session_id，存储在内存中的`Session`表中（key为session_id，value为过期时间戳）。服务端在HTTP响应头中写入`Set-Cookie: session_id=<uuid>; Max-Age=3600; HttpOnly`。后续管理员访问`/api/admin/*`接口时，服务端从请求Cookie中提取session_id，查询Session表验证其有效性和过期时间。

密码哈希方面，`PasswordHasher::hash()`使用libsodium的`crypto_pwhash()`函数，算法参数配置为：OpsLimit = `crypto_pwhash_OPSLIMIT_INTERACTIVE`（约2次迭代），MemLimit = `crypto_pwhash_MEMLIMIT_INTERACTIVE`（约64MB内存占用）。`PasswordHasher::verify()`使用`crypto_pwhash_verify()`进行验证。

#### 5.3.4 Web端管理接口

**POST /api/admin/login**：管理员登录。验证用户名和密码，成功后设置Session Cookie。

**GET /api/admin/field/list**：查询所有瓜田列表。返回所有`field_production`表中的`field_id`。

**GET /api/admin/watermelon/list?field_id=1001**：查询指定瓜田下所有西瓜的最新数据。返回各西瓜的`device_id`、最新糖度、最新成熟度、采集时间等信息。

**GET /api/admin/watermelon/history?device_id=1001-01-01**：查询指定西瓜近84天的糖度历史数据。返回时间序列数据用于绘制折线图。

**GET /api/admin/field/environment?field_id=1001**：查询指定瓜田近84天的环境数据（温度/湿度/光照）。返回时间序列数据用于绘制折线图。

### 5.4 高并发场景适配设计

本系统的后端服务虽然采用单线程顺序处理的轻量级设计，但在高并发场景下仍具备一定的处理能力，主要通过以下设计实现：

**httplib的线程池支持**：cpp-httplib内置了线程池支持，可以在创建Server时通过参数配置线程池大小，默认情况下httplib会为每个请求分配一个线程处理，充分利用多核CPU的并行处理能力。

**MySQL单连接设计**：当前版本使用单MySQL连接+互斥锁（`std::mutex mtx`）保护所有数据库操作。虽然这限制了数据库层面的并发度，但对于本系统预期的接入规模（数十到数百台设备）而言，单连接已足够支撑。

**断网数据补发的错峰设计**：设备端在WiFi恢复后采用随机延迟（0～10秒）补发策略，避免大量设备同时发起补发请求导致服务器瞬时压力过载。

### 5.5 本章小结

本章详细介绍了后端数据处理中心的设计与实现。

开发环境方面，采用C++语言和CMake构建系统，集成cpp-httplib（HTTP框架）、nlohmann/json（JSON处理）和libsodium（密码学）三个核心第三方库，选用MySQL 8.0作为数据库。

系统架构方面，采用控制层→服务层→数据访问层的三层架构，职责分离清晰。

数据库设计方面，详细设计了5张核心数据表：管理员信息表（admin_users）、设备校验表（device_auth）、瓜田生产信息表（field_production）、西瓜信息表（watermelon_data）和瓜田环境信息表（field_environment），明确了各表的主键、索引和业务规则。

核心API方面，详细阐述了设备校时接口（GET /api/device/time）和数据上传接口（POST /api/device/upload）的请求格式、响应格式和完整的业务处理流程。数据上传接口的核心处理流程包括：身份认证→数据解析→时间戳获取→瓜田号提取→数据校验→糖度计算→成熟度计算→双表入库→响应返回。

身份校验方面，实现了设备Token认证（device_auth表+状态检查）和管理员Session认证（内存Session表+Cookie）两套机制，密码存储使用libsodium的Argon2id强哈希算法。

### 7.2 系统不足与未来展望

尽管本系统已实现了完整的功能并通过了全面测试，但仍存在以下不足之处，有待后续研究改进：

**（1）糖度预测模型的精度和泛化能力有待提升**

当前MLR模型的回归系数为初步估算值，未经严格的实验标定，检测精度（平均误差0.36°Brix）与专业光谱仪（误差<0.1°Brix）仍有差距。未来可考虑以下改进方向：开展大规模的西瓜样本标定实验（建议样本量≥100个，覆盖不同品种、产地、成熟度），建立更加准确可靠的预测模型；引入偏最小二乘回归（PLSR）、支持向量回归（SVR）或深度学习等更先进的建模方法，提升模型的预测精度和泛化能力；建立模型的在线更新机制，允许在部署后通过新样本持续优化模型参数。

**（2）设备功耗优化空间较大**

当前设备在工作状态下功耗约250mA@5V（1.25W），主要耗电模块包括ESP32主控（80mA）、TFT屏幕背光（40mA）、LED光源（700mA，但仅在采集时点亮）和AS7341传感器（20mA）。未来可考虑以下改进：引入Deep-Sleep低功耗模式，在非采集期间让ESP32进入休眠状态，降低平均功耗；优化TFT背光亮度或采用OLED显示屏替代TFT，降低显示功耗；实现电池供电方案（锂电池+TP4056充电管理），提升设备的移动性和便携性。

**（3）通信方式单一**

当前系统仅支持WiFi一种通信方式。在没有WiFi覆盖的偏远田间或室内封闭环境，设备无法正常工作。未来可考虑增加以下通信方式：蓝牙BLE低功耗蓝牙，可与手机直接通信，适合近距离数据传输；NB-IoT/LoRa远距离低功耗广域网，适合偏远田间的物联网应用；NFC/RFID近场通信，可用于西瓜的快速身份识别和溯源。

**（4）Web管理平台功能有待完善**

当前Web平台仅提供了数据查看和可视化展示功能，缺少设备管理、数据导出、报警规则配置等高级功能。未来可扩展以下功能：设备注册与分组管理，支持新设备的添加和配置；数据导出功能，支持CSV/Excel格式的历史数据导出；报警规则配置，当糖度低于阈值时发送邮件或短信通知；多用户权限管理，支持不同角色（管理员、农技师、采购商）的差异化权限控制。

**（5）多品类水果检测功能**

当前系统仅针对西瓜设计了糖度预测模型。未来可将系统扩展为多品类果蔬糖度检测平台，通过更换预测模型参数或加载不同模型文件，支持苹果、芒果、草莓、葡萄等多种水果的糖度检测。

**（6）多设备组网管理**

当前系统以单设备独立工作模式为主。未来可研究多设备组网协同工作机制，支持一台网关设备连接管理多个检测终端，实现多瓜田数据的集中采集、统一管理和协同分析。

**B.1 糖度计算算法（sugar_calc.h）**

```cpp
#pragma once
#include <nlohmann/json.hpp>

namespace SugarCalc {
    double calculate(const nlohmann::json& spectrum) {
        // MLR模型参数（需实验标定）
        constexpr double k = 5.2;   // 截距
        constexpr double a = 0.0012; // ch415系数
        constexpr double b = 0.0005; // ch445系数
        constexpr double c = 0.0021; // ch480系数
        
        double ch415 = spectrum.value("ch415", 0);
        double ch445 = spectrum.value("ch445", 0);
        double ch480 = spectrum.value("ch480", 0);
        
        double brix = a * ch415 + b * ch445 + c * ch480 + k;
        
        // 合理性约束
        if (brix < 0)   return 0.0;
        if (brix > 20)  return 20.0;
        
        return brix;
    }
}
```

```cpp
void loop() {
    unsigned long now = millis();
    
    // WiFi后台维护与离线补发
    NetworkManager::maintainWiFi();
    bool currentWiFiState = (WiFi.status() == WL_CONNECTED);
    
    // WiFi恢复检测 → 启动补发状态机
    if (currentWiFiState && !wasWiFiConnected) {
        isRecoveringData = true;
        recoverStartTime = now;
        recoverDelayDelay = random(0, 10000); // 随机0~10秒错峰
    }
    wasWiFiConnected = currentWiFiState;
    
    // 离线数据补发
    if (isRecoveringData && currentWiFiState && 
        (now - recoverStartTime > recoverDelayDelay)) {
        // 查找未上传记录 → 上传 → 更新状态 → UI刷新
        if (StorageManager::getInstance().getUnuploadedRecord(...)) {
            NetworkManager::uploadData(...);
            StorageManager::getInstance().markAsUploaded(target_ts);
            ui.markIndustrialHistoryUploaded(timeStr.c_str());
        } else {
            isRecoveringData = false;
        }
    }
    
    // 工业模式定时采集
    if (sysMode == MODE_INDUSTRIAL) {
        if (now - lastEnvUpdate > 2000) {
            sensorMgr.readEnvironment(t, h, l);
            ui.updateEnv(t, h, l);
        }
        
        unsigned long elapsed = now - lastAutoMeasure;
        if (elapsed >= 300000) { // 5分钟
            onMeasureBtnClicked();
            lastAutoMeasure = now;
        }
    }
    
    lv_timer_handler();
    delay(5);
}
```

```cpp
double SugarCalc::calculate(const nlohmann::json& spectrum) {
    double ch415 = spectrum.value("ch415", 0);
    double ch445 = spectrum.value("ch445", 0);
    double ch480 = spectrum.value("ch480", 0);
    
    double brix = 0.0012 * ch415 + 0.0005 * ch445 + 0.0021 * ch480 + 5.2;
    
    if (brix < 0) return 0.0;
    if (brix > 20.0) return 20.0;
    return brix;
}
```
