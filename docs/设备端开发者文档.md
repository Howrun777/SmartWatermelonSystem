# 设备端开发者文档

本文档面向负责 ESP32 设备端开发、维护的工程师，描述系统的功能、设计决策、代码结构和通信协议。所有描述与 `src/` 目录下实际代码严格对应。

---

## 1 系统概述

### 1.1 项目定位

设备端是整个 SmartWatermelonSystem 的数据采集入口，负责西瓜表面光谱数据采集、糖度本地预测、人机交互界面展示和检测数据的网络上传。系统面向两种使用场景：

- **Consumer Test Mode（测试模式）**：面向消费者或收购商，无需网络，内存存储，最多20条，纯现场快速检测。
- **Industrial Mode（工业模式）**：面向规模化种植采集，LittleFS 持久化存储，HTTP 上传服务器，支持离线断网，WiFi 恢复后自动补发。

### 1.2 技术栈

| 层级 | 技术选型 | 版本 |
|------|---------|------|
| 主控 | ESP32 DevKit V1 (Xtensa dual-core 240MHz) | — |
| 开发框架 | Arduino-ESP32 (PlatformIO) | — |
| 图形库 | LVGL | 8.3.9 |
| 显示屏驱动 | TFT_eSPI | 2.5.31 |
| 触摸驱动 | XPT2046_Touchscreen | (git) |
| JSON | ArduinoJson | 6.21.3 |
| 文件系统 | LittleFS (ESP32 内置 Flash 分区) | — |
| 网络 | WiFi 802.11b/g/n + HTTPClient | — |

## 3 核心模块详解

### 3.1 主程序入口（main.cpp）

#### 3.1.1 初始化顺序（setup）

```
Serial.begin(115200)
  → SensorManager.begin()       // I2C 总线初始化 + 各传感器检测
  → RTCManager.begin()          // DS3231 检测，丢失电力则等待服务器校时
  → StorageManager.begin()      // 挂载 LittleFS，容量检查
  → TFT init + LVGL init       // 显示和触摸初始化
  → UIManager.init()           // 注册 4 个回调，绘制启动页
  → NetworkManager.connectWiFi() → fetchServerTime() → RTCManager.syncTime()
  → showModeSelection()         // 模式选择按钮出现
```

#### 3.1.2 主循环（loop）

```
loop() 每轮执行：
  1. NetworkManager::maintainWiFi()           // 断线时每30秒重连一次
  2. 检测 WiFi 从断开→恢复的边沿信号         // 触发补发状态机
  3. 若处于补发状态机：查找 up=0 的最老记录 → 上传 → 标记 up=1 → UI 刷新
  4. 若为工业模式：
     - 每 ENV_UPDATE_INTERVAL(2000ms) 刷新环境数据显示
     - 每 AUTO_UPLOAD_INTERVAL(300000ms) 自动触发一次检测
  5. lv_timer_handler()                      // LVGL 渲染
  6. delay(5)
```

### 3.2 传感器模块（SensorManager）

#### 3.2.1 I2C 总线配置

I2C 总线固定在 GPIO27 (SDA) + GPIO22 (SCL)，所有 I2C 传感器并联在这组总线上。各传感器 I2C 地址：

| 传感器 | 地址 | 用途 | 可选 |
|--------|------|------|------|
| AS7341 | 0x39 | 光谱数据采集（核心） | **必选** |
| SHT31 | 0x44 | 温度/湿度 | 可选 |
| BH1750 | 0x23 | 光照强度 | 可选 |

#### 3.2.2 传感器解耦

启动时通过 I2C 应答检测传感器是否存在（`has_as7341 / has_sht31 / has_bh1750`）。缺失的传感器不影响核心功能：

- SHT31 缺失：温度=99.0，湿度=99.0
- BH1750 缺失：光照=99
- AS7341 缺失：`readSpectrum()` 返回 false，测量流程中止

#### 3.2.3 光谱采集流程

```
1. controlLight(true)   → GPIO26 输出 HIGH，LED 点亮
2. delay(500)          → 等待 LED 光谱稳定
3. as7341.readAllChannels()
4. 读取 10 个通道：ch415/ch445/ch480/ch515/ch555/ch595/ch640/ch680/Clear/NIR
5. controlLight(false)  → GPIO26 输出 LOW，LED 熄灭
```

AS7341 配置：ATIME=50，ASTEP=999，GAIN=256X。单次积分时间约 50×1000×2.78μs = 139ms。

### 3.3 实时时钟（RTCManager）

#### 3.3.1 双校时机制

RTCManager 采用单例模式，统一管理 DS3231 和 ESP32 系统时钟：

```
DS3231 优先：
  设备上电 → RTC.begin() → 检测 DS3231 是否丢失电力
    ├─ 丢失电力（lostPower()=true）：等待服务器校时
    └─ 正常：syncTime(rtc.now().unixtime()) → 同步到 ESP32 系统时钟

网络校准（双节点）：
  ├─ setup() 中：WiFi 连接后调用 /api/device/time → syncTime(server_time)
  └─ 每次上传成功后：响应中的 server_time → syncTime(server_time)
```

#### 3.3.2 北京时间显示

`formatTime(ts)` 对 UTC 时间戳强制 +8 小时偏移（`ts + 8*3600`），确保屏幕显示北京时间。

#### 3.3.3 无 DS3231 的降级方案

若 DS3231 通信失败（`has_ds3231=false`），`getTimestamp()` 回退到 `gettimeofday()` 获取 ESP32 内部时钟。ESP32 断电后内部时钟归零，需依赖下次服务器校时恢复。

### 3.4 网络模块（HttpClient）

#### 3.4.1 WiFi 连接策略

`connectWiFi()`：最多等待 20 次 × 500ms = 10 秒，超时则进入离线模式，不阻塞 UI。

`maintainWiFi()`：仅在 `WiFi.status() != WL_CONNECTED` 时触发重连尝试，两次尝试间隔 30 秒，采用非阻塞方式（`WiFi.begin()` 后直接返回），避免阻塞 `lv_timer_handler()`。

#### 3.4.2 离线补发（关键机制）

```
触发条件：WiFi 从断开变为连接（边沿检测）

步骤：
  1. 记录当前时间 recoverStartTime
  2. 随机等待 0~10 秒（recoverDelayDelay）← 防雪崩
  3. 等待结束后，从 records.jsonl 顺序查找第一条 up=0 记录
  4. 构造 payload 并 POST /api/device/upload
  5. 成功 → markAsUploaded(ts) → UI 表格该行 Up 列从 N 变 Y
  6. 重复步骤 3~5，直至没有 up=0 记录 → isRecoveringData=false
```

补发结束后，新采集的数据立即上传，不影响实时数据流。

#### 3.4.3 时间同步接口路径

时间同步接口由 `SERVER_URL` 自动构造：将 `/device/upload` 替换为 `/device/time`。响应格式：

```json
{"code": 200, "data": {"server_time": 1712345678}}
```

### 3.5 存储模块（StorageManager）

#### 3.5.1 存储文件

路径：`/records.jsonl`（LittleFS 分区）

格式：JSON Lines（每行一个完整 JSON 对象，以 `\n` 分隔）

单条记录结构：
```json
{"ts":1712345678,"brix":12.5,"t":25.3,"h":60.2,"l":10000,"up":0,"spec":{...}}
```

字段含义：`ts`=采集时间戳，`brix`=糖度，`t`/`h`/`l`=环境数据，`up`=上传状态（1=Y, 0=N），`spec`=光谱数据。

#### 3.5.2 容量管理

`checkAndCleanCapacity()` 在以下时机被调用：
- 设备启动时（`StorageManager::begin()`）
- 每次 `saveRecord()` 之后

清理条件：已用空间 > 总容量 × 80%

清理策略：删除文件中最早 50% 的记录（`linesToSkip = totalLines / 2`），保留较新的数据。

#### 3.5.3 标记与查询

`markAsUploaded(target_ts)`：遍历文件，找到 `ts` 匹配的那条记录，将 `up` 从 0 改为 1，写入临时文件后 rename 替换原文件。

`getUnuploadedRecord()`：顺序遍历文件，返回第一条 `up=0` 的记录（按文件顺序，即最老的未上传记录优先补发）。

### 3.6 双模式业务逻辑

#### 3.6.1 模式定义

```cpp
enum SystemMode {
    MODE_WAITING = 0,     // 等待时间同步
    MODE_TEST = 1,        // 测试模式
    MODE_INDUSTRIAL = 2   // 工业模式
};
```

#### 3.6.2 测试模式（Consumer Test Mode）

- 环境数据不采集，强制设为 99
- 数据存于 `test_sugar_sum` / `test_count`（内存），`MAX_TEST_COUNT = 20`
- 超过 20 条自动清空（`onClearTestClicked()`）
- 不上传服务器
- 不写入 LittleFS

#### 3.6.3 工业模式（Industrial Mode）

- 采集并上传光谱 + 环境数据
- 每次测量后写入 `records.jsonl`（标记 up=1 或 up=0）
- 5 分钟定时自动触发测量（`AUTO_UPLOAD_INTERVAL = 300000ms`）
- 支持手动 MEASURE 按钮，重置自动倒计时

#### 3.6.4 双模式对比

| 特性 | 测试模式 | 工业模式 |
|------|---------|---------|
| 网络依赖 | 无 | 弱（可离线） |
| 数据存储 | 内存（≤20条） | LittleFS 持久化 |
| 环境数据 | 不采集（=99） | 采集 |
| 数据上传 | 不上传 | 每次上传 |
| 离线补发 | 不支持 | 支持 |
| 自动定时 | 不支持 | 5分钟间隔 |
| 适用场景 | 商超/田头快速检测 | 规模化种植数据采集 |

### 3.7 糖度算法（sugar_calc.h）

```cpp
float brix = 0.0012 * ch415 + 0.0005 * ch445 + 0.0021 * ch480 + 5.2;
if (brix < 0)   return 0.0;
if (brix > 20)  return 20.0;
return brix;
```

基于 ch415/ch445/ch480 三个通道的多元线性回归（MLR）模型，系数需实验标定。结果约束在 0~20°Brix 范围内。设备端算法与后端 `SugarCalc.cpp` 保持一致。

## 5 通信协议

### 5.1 数据上传

**接口**：`POST /api/device/upload`

**请求头**：

```
Content-Type: application/json
device_id: 1001-01-01
token: device-token-001
```

**请求体**：

```json
{
  "spectrum_json": {
    "ch415": 123, "ch445": 456, "ch480": 789,
    "ch515": 234, "ch555": 567, "ch595": 890,
    "ch640": 321, "ch680": 654, "chClear": 999, "chNIR": 111
  },
  "temperature": 25.5,
  "humidity": 60.2,
  "light": 10000,
  "collected_at": 1712345678
}
```

`collected_at` 为可选字段。若不填，服务器以自身收到请求的时间作为采集时间戳。

**成功响应**（HTTP 200）：

```json
{
  "code": 200,
  "msg": "数据上传成功",
  "data": {
    "sugar_brix": 12.5,
    "maturity_score": 1.02,
    "collected_at": 1712345678,
    "server_time": 1712345679
  }
}
```

**失败响应**：

| 状态码 | 含义 | 处理方式 |
|--------|------|---------|
| 401 | 设备认证失败（device_id/token 不匹配或 status=0） | 进入离线模式 |
| 400 | 数据校验失败（瓜田不存在等） | 记录错误信息到 status |
| 500 | 服务器内部错误 | 进入离线模式 |

### 5.2 时间同步

**接口**：`GET /api/device/time`

**请求头**：同数据上传。

**响应**：

```json
{"code": 200, "data": {"server_time": 1712345678}}
```

## 6 启动与部署

### 6.1 开发环境

```bash
# 安装 PlatformIO（VS Code 插件或 CLI）
# 项目根目录执行
pio run --target upload      # 编译并烧录
pio run --target monitor     # 打开串口监视器（115200 baud）
pio run --target uploadfs    # 烧录 LittleFS 分区（首次或文件系统修改后）
```

### 6.2 首次部署步骤

1. 修改 `config.h` 中的 `DEVICE_ID`、`DEVICE_TOKEN`（需与后端 `device_auth` 表一致）、WiFi 凭据、`SERVER_URL`
2. `pio run --target upload` 烧录固件
3. `pio run --target uploadfs` 烧录 LittleFS 文件系统（首次必需）
4. `pio run --target monitor` 确认启动日志正常（传感器检测、WiFi 连接、LVGL 初始化）
5. 确认服务器端 `device_auth` 表中已注册该 `DEVICE_ID` 和 `token`

### 6.3 服务器依赖

设备正常工作依赖后端服务器（`FieldDataProcessingServer`）处于运行状态：

- `POST /api/device/upload`（数据上传）
- `GET /api/device/time`（时间同步）

服务器地址在 `config.h` 中配置。设备可在服务器不可达时独立工作（测试模式 / 离线存储）。

## 7 维护指南

### 7.1 常见问题排查

| 现象 | 排查方向 |
|------|---------|
| 启动后屏幕不亮 | 检查 TFT_BL 引脚（platformio.ini 中 `TFT_BL`）是否为 3.3V；检查 LVGL `my_disp_flush` 回调是否注册 |
| 触摸无响应 | 检查 XPT2046 接线（MOSI/MISO/CLK/CS/IRQ）；检查 `ts.touched()` 始终返回 false 时的 ADC 值范围（map 映射可能需调整） |
| AS7341 检测失败 | 确认 I2C 地址为 0x39；确认 SDA=GPIO27, SCL=GPIO22；检查传感器供电 3.3V |
| WiFi 无法连接 | 确认 SSID/密码正确；确认 WiFi 路由器工作正常；检查 `WiFi.status()` 返回值 |
| 数据无法上传但无报错 | 确认 `device_auth` 表中 `status=1`；确认 `DEVICE_ID` 和 `DEVICE_TOKEN` 与数据库一致 |
| 离线记录不补发 | 检查 `records.jsonl` 中是否存在 `up=0` 记录；检查 `markAsUploaded()` 时间戳匹配逻辑 |
| 工业模式 UI 无环境数据 | 确认 SHT31 和 BH1750 传感器 I2C 地址无冲突；确认 `has_sht31`/`has_bh1750` 检测为 true |

### 7.2 关键日志关键词

通过 `pio run --target monitor` 观察以下日志判断系统状态：

| 日志 | 含义 |
|------|------|
| `DS3231 RTC Init Success!` | DS3231 检测成功 |
| `DS3231 RTC Not Found!` | DS3231 未检测到，降级使用 ESP32 内部时钟 |
| `DS3231 lost power, waiting for NTP/Server sync...` | DS3231 电池耗尽或首次使用，需服务器校时 |
| `DS3231 & System time synced to Server Time!` | 双时钟同步成功 |
| `WiFi Connected! IP: ...` | WiFi 连接成功 |
| `WiFi Failed. Entering Offline Mode.` | WiFi 连接超时，进入离线模式 |
| `Record saved to disk. Uploaded: Y/N` | 数据已存入 records.jsonl |
| `Storage almost full! Triggering 50% cleanup...` | 存储容量不足，自动清理中 |

### 7.3 存储数据导出

将 ESP32 的 LittleFS 分区通过 PlatformIO 导出到本地：

```bash
pio run --target uploadfs  # 烧录文件系统到设备
pio run --target readfs    # （部分版本支持）读取 LittleFS 内容
```

如需手动检查 `records.jsonl` 内容，可通过 Arduino 代码或 ESP-IDF 的 `esp_littlefs` 工具访问分区。

### 7.4 糖度模型更新

如需更换糖度预测模型（调整系数或算法），只需修改 `src/algorithm/sugar_calc.h` 中的 `calculate()` 方法，**同步修改后端** `FieldDataProcessingServer/src/data_process/SugarCalc.cpp` 中对应的系数，确保设备端与服务器端的计算结果一致。

## 8 硬件引脚速查

### 8.1 TFT 显示屏引脚

在 `platformio.ini` 的 `build_flags` 中定义，TFT_eSPI 库直接读取：

| 符号 | 值 | 说明 |
|------|----|------|
| TFT_MISO | 12 | SPI 数据输入 |
