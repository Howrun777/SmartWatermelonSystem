# SmartWatermelonSystem 开发者文档

本文档面向项目维护、二次开发、部署运维、毕业设计答辩和后续模型标定。内容以当前仓库实现为准，覆盖 ESP32 设备端、C++ 后端、Web 前端、MySQL 数据库、HTTP API、离线补发、光谱数据导出、运维命令和常见故障定位。

---

## 1. 项目概述

SmartWatermelonSystem 是一个面向西瓜无损糖度检测的端到端系统。设备端使用 ESP32 采集 AS7341 多光谱数据和瓜田环境数据，服务端负责设备认证、糖度计算、成熟度计算和 MySQL 入库，Web 前端用于后台可视化展示。

系统核心目标：

- 使用 AS7341 多光谱传感器采集西瓜表皮反射数据。
- 使用固定光源 MLR 模型估算 Brix 糖度。
- 支持温湿度、光照等环境数据同步采集。
- 支持设备端断网缓存、恢复联网后补发。
- 支持服务端设备认证、数据校验、自动重连数据库。
- 支持 Web 后台查看瓜田、西瓜实时糖度、历史曲线和设备状态。
- 支持导出真实糖度标注数据，用于后续模型重训。

当前项目定位是毕业设计级原型系统，但已经具备基本工程化能力：设备端缓存、服务端认证、Web 可视化、数据库持久化和真实样本数据归档。

## 2. 总体架构

```text
ESP32 设备端
  ├─ 采集 AS7341 多光谱
  ├─ 采集 SHT31 温湿度
  ├─ 采集 BH1750 光照
  ├─ 使用 DS3231 / 服务端授时保证时间戳

  ├─ 校验瓜田是否存在
  ├─ 使用服务端 SugarCalc 计算糖度
  ├─ 使用 mature_sugar_threshold 计算成熟度
  ├─ 写入 sws_db.watermelon_data
  ├─ 写入 sws_db.field_environment
  └─ 托管 Web 静态页面和管理接口

MySQL 数据库
  ├─ admin_users
  ├─ device_auth
  ├─ field_production
  ├─ watermelon_data
  └─ field_environment

Web 前端
  ├─ 登录页 /admin/login.html
  ├─ 监控大屏 /admin/dashboard.html
  ├─ 获取瓜田列表
  ├─ 获取西瓜实时数据
  ├─ 获取单瓜历史糖度
  └─ 获取瓜田环境历史
```

## 5. 设备端开发说明

### 5.1 配置文件

设备端核心配置位于：

```text
FieldAcquisitionTerminal/config.h
```

关键配置：

```cpp
#define DEVICE_ID "1001-01-01"
#define WATERMELON_VARIETY "Kirin Watermelon"
#define DEVICE_TOKEN "device-token-001"

#define WIFI_SSID "..."
#define WIFI_PASS "..."
#define SERVER_URL "http://47.107.41.102:8080/api/device/upload"

#define I2C_SDA_PIN 27
#define I2C_SCL_PIN 22

#define ENV_UPDATE_INTERVAL 2000
#define AUTO_UPLOAD_INTERVAL 300000
```

注意事项：

- `DEVICE_ID` 格式为 `瓜田号-组号-瓜号`，例如 `1001-01-01`。
- 服务端通过 `device_id.substr(0, 4)` 得到 `field_id`。
- `DEVICE_TOKEN` 必须和数据库 `device_auth.token` 一致。
- `SERVER_URL` 必须指向当前运行的后端 `/api/device/upload`。
- 如果换设备或换瓜田，必须同步修改设备端配置和数据库设备认证表。

### 5.2 设备启动流程

设备端主流程在：

```text
FieldAcquisitionTerminal/src/main.cpp
```

启动流程：

```text
setup()
  ├─ 初始化串口
  ├─ 初始化蜂鸣器
  ├─ 初始化 LittleFS
  ├─ 初始化传感器
  ├─ 初始化 TFT / LVGL / 触摸
  ├─ 读取 RTC 时间
  ├─ WiFi 可用时请求 /api/device/time 授时
  └─ 时间合法后进入工业模式
```

时间有效性一般以 Unix 时间戳大于 `1600000000` 作为判断，避免出现 1970 年数据。

### 5.3 采集流程

工业模式下单次采集的逻辑：

```text
用户触发或定时触发采集
  ├─ 读取 AS7341 光谱
  ├─ 读取温湿度
  ├─ 读取光照
  ├─ 获取当前时间戳 collected_at
  ├─ 设备端本地 SugarCalc 计算糖度
  ├─ 组装 JSON
  ├─ HTTP POST 上传
  ├─ 上传成功则本地标记 up=1
  ├─ 上传失败则本地标记 up=0
  └─ 更新屏幕历史表
```

上传 JSON 示例：

```json
{
  "collected_at": 1779197226,
  "temperature": 25.1,
  "humidity": 58.3,
  "light": 5,
  "spectrum_json": {
    "ch415": 3214,
    "ch445": 3922,
    "ch480": 4831,
    "ch515": 12171,
    "ch555": 17804,
    "ch595": 29421,
    "ch640": 23915,
    "ch680": 9719,
    "chClear": 51000,
    "chNIR": 5386
  }
}
```

### 5.4 离线缓存

设备端离线缓存由 `StorageManager` 管理：

```text
FieldAcquisitionTerminal/src/storage/StorageManager.cpp
```

本地文件：

```text
/records.jsonl
```

单行结构：

```json
{
  "ts": 1779197226,
  "brix": 11.5,
  "t": 25.1,
  "h": 58.3,
  "l": 5,
  "up": 0,
  "spec": {
    "ch415": 3214,
    "ch445": 3922
  }
}
```

关键逻辑：

- 断网时保存 `up=0`。
- 联网后查找 `up=0` 的记录补发。
- 补发成功后调用 `markAsUploaded(ts)`。
- 秒级时间戳重复时，保存逻辑会尝试偏移时间戳，避免数据库主键冲突。
- 存储空间不足时会清理一部分旧记录。

### 5.5 设备上传成功判定

设备端不能只看 HTTP 状态码。当前后端业务失败也可能返回 HTTP 200，因此设备端必须同时检查：

```text
HTTP status == 200
JSON code == 200
```

正确逻辑已经在：

```text
FieldAcquisitionTerminal/src/network/HttpClient.cpp
```

### 6.3 设备上传处理流程

`POST /api/device/upload` 的核心流程：

```text
读取 Header device_id/token
  ├─ DeviceAuth::authenticate()
  ├─ json::parse(req.body)
  ├─ 读取 temperature/humidity/light/spectrum_json
  ├─ 读取 collected_at，缺失时使用服务器时间
  ├─ 从 device_id 前 4 位提取 field_id
  ├─ DataCheck::isFieldExist(field_id)
  ├─ SugarCalc::calculate(spectrum)
  ├─ MaturityCalc::calculate(field_id, sugar_brix)
  ├─ INSERT watermelon_data
  ├─ 环境有效时 INSERT field_environment
  └─ 返回 code/msg/data
```

认证失败响应：

```json
{
  "code": 401,
  "msg": "认证失败：Token错误、设备不存在或已被禁用"
}
```

成功响应：

```json
{
  "code": 200,
  "msg": "数据上传成功",
  "data": {
    "sugar_brix": 14.66,
    "maturity_score": 1.173,
    "server_time": 1779180848
  }
}
```

### 6.4 MySQL 驱动

数据库封装在：

```text
FieldDataProcessingServer/src/db/MySQLDriver.cpp
```

当前能力：

- 单例连接。
- `std::mutex` 串行化查询和写入。
- `mysql_real_escape_string` 转义外部字符串。
- `execute()` 执行写入 SQL。
- `query()` 返回 `vector<map<string,string>>`。
- 每次执行前调用 `ensureConnected()`。
- 若 MySQL 长连接断开，自动关闭旧连接并重连。

自动重连非常重要。曾经出现过后端进程仍在监听 8080，但 MySQL 连接已经从 `SHOW PROCESSLIST` 消失，导致认证查询失败并返回 401。当前驱动已补上 `mysql_ping()` 检测和重连逻辑。

### 6.5 糖度计算

服务端糖度计算在：

```text
FieldDataProcessingServer/src/data_process/SugarCalc.cpp
```

当前使用固定光源原始通道 MLR：

```text
X445 = ch445 / 10000
X480 = ch480 / 10000
X680 = ch680 / 10000
XNIR = chNIR / 10000

raw_brix =
  13.21941763
  - 23.50752069 * X445
  +  8.59055811 * X480
  +  5.98831600 * X680
  -  8.41861824 * XNIR
```

边界处理：

- 任一关键通道低于 `10`，返回 `0.0`。
- 结果为 NaN，返回 `0.0`。
- 小于 `0`，截断为 `0`。
- 大于 `20`，截断为 `20`。

`chClear` 当前只用于诊断，不参与模型计算。原因是固定光源结构下 `chClear` 很容易达到 `51000` 饱和，作为归一化分母会放大误差。

### 6.6 成熟度计算

成熟度计算在：

```text
FieldDataProcessingServer/src/data_process/MaturityCalc.cpp
```

公式：

```text
maturity_score = sugar_brix / mature_sugar_threshold
```

成熟糖度阈值来自：

```sql
SELECT mature_sugar_threshold
FROM field_production
WHERE field_id = ?
```

示例：

```text
sugar_brix = 13.0
mature_sugar_threshold = 12.5
maturity_score = 1.04
```

### 6.7 环境数据校验

环境校验在：

```text
FieldDataProcessingServer/src/data_process/DataCheck.cpp
```

当前规则：

```text
temperature == 99.0
humidity == 99.0
light == 99
```

三者同时为哨兵值时，认为环境传感器缺失，跳过 `field_environment` 写入，但不影响西瓜光谱数据入库。

### 8.3 POST /api/device/upload

用途：上传西瓜光谱和环境数据。

请求头：

```text
Content-Type: application/json
device_id: 1001-01-01
token: device-token-001
```

请求体：

```json
{
  "collected_at": 1779197226,
  "temperature": 25.1,
  "humidity": 58.3,
  "light": 5,
  "spectrum_json": {
    "ch415": 3214,
    "ch445": 3922,
    "ch480": 4831,
    "ch515": 12171,
    "ch555": 17804,
    "ch595": 29421,
    "ch640": 23915,
    "ch680": 9719,
    "chClear": 51000,
    "chNIR": 5386
  }
}
```

成功响应：

```json
{
  "code": 200,
  "msg": "数据上传成功",
  "data": {
    "sugar_brix": 15.2,
    "maturity_score": 1.216,
    "server_time": 1779197228
  }
}
```

常见失败：

| code | 原因 |
| --- | --- |
| 400 | JSON 格式错误、字段异常、瓜田未注册 |
| 401 | Token 错误、设备不存在、设备禁用 |
| 500 | 数据库写入失败，常见于重复主键 |

注意：当前后端业务失败也通过 HTTP 200 返回 JSON，因此调用方必须检查 JSON `code`。

## 10. 光谱数据采集与标定

### 10.1 当前真实数据文件

当前 docs 中已有多组真实 Brix 数据：

```text
docs/西瓜光谱数据-6.8.txt
docs/西瓜光谱数据-8.7.txt
docs/西瓜光谱数据-9.3.txt
docs/西瓜光谱数据-11.5.txt
```

文件格式：

```text
device_id	collected_at	true_brix	server_brix	ch415	ch445	ch480	ch515	ch555	ch595	ch640	ch680	chClear	chNIR
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| device_id | 设备编号 |
| collected_at | 采集时间戳 |
| true_brix | 真实糖度，来自人工糖度计标注 |
| server_brix | 服务端当前模型计算结果 |
| ch415 ~ ch680 | AS7341 可见光通道 |
| chClear | Clear 通道 |
| chNIR | 近红外通道 |

### 10.2 数据采集规范

为保证模型可用性，采集时应尽量固定以下条件：

- 光源功率固定。
- 光源到西瓜距离固定。
- 传感器到西瓜距离固定。
- 传感器角度固定。
- 每个西瓜采集时尽量选择同类测点。
- 条纹区域和非条纹区域不要混合建模。
- 每个真实糖度至少采集几十条。
- 每个真实糖度最好来自多个西瓜，而不是一个西瓜多次重复。

### 10.3 质量风险

以下情况会明显影响糖度模型：

- 多个通道达到 `51000`，说明通道饱和。
- 光源未开启，光谱整体接近 0。
- 测点落在黑色条纹上。
- 西瓜表皮湿润、反光或角度变化大。
- 同一真实 Brix 标签下只有单个西瓜，容易学到个体差异。

### 10.4 导出某一真实糖度数据

以 `11.5Brix` 为例：

```bash
mysql -h127.0.0.1 -usws_user -p --batch --raw sws_db -e "
SELECT
  device_id,
  collected_at,
  11.5 AS true_brix,
  sugar_brix AS server_brix,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch415')) AS ch415,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch445')) AS ch445,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch480')) AS ch480,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch515')) AS ch515,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch555')) AS ch555,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch595')) AS ch595,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch640')) AS ch640,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.ch680')) AS ch680,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.chClear')) AS chClear,
  JSON_UNQUOTE(JSON_EXTRACT(spectrum_json, '$.chNIR')) AS chNIR
FROM watermelon_data
ORDER BY collected_at ASC;
" > /home/Howrun1/SmartWatermelonSystem/docs/西瓜光谱数据-11.5.txt
```

### 10.5 清空当前采集数据

清空当前西瓜数据，同时删除对应时间戳的环境数据：


## 13. 常见故障排查

### 13.1 设备显示上传成功，但数据库没有数据

检查顺序：

1. 确认后端正在监听：

```bash
ss -ltnp | grep ':8080'
```

2. 确认设备访问的是当前服务器：

```bash
curl http://47.107.41.102:8080/ping
```

3. 确认设备认证表：

```sql
SELECT * FROM device_auth WHERE device_id='1001-01-01';
```

4. 确认后端 MySQL 连接是否存在：

```sql
SHOW PROCESSLIST;
```

当前 `MySQLDriver` 已具备自动重连能力。如果运行的是旧二进制，应重新编译并重启服务。

### 13.2 上传返回 401

可能原因：

- Header 没有 `device_id`。
- Header 没有 `token`。
- `device_auth` 中没有该设备。
- Token 不一致。
- `status=0`。
- 后端旧版本 MySQL 长连接断开，导致认证查询失败。

排查：

```bash
mysql -h127.0.0.1 -usws_user -p sws_db -e "
SELECT device_id, token, status
FROM device_auth
WHERE device_id='1001-01-01';
"
```

### 13.3 上传返回 500

常见原因：

- `(device_id, collected_at)` 主键重复。
- 环境表 `(field_id, collected_at)` 主键重复。
- MySQL 权限不足。
- JSON 字段不合法。

处理：

- 检查设备端时间戳是否重复。
- 检查离线补发记录是否重复。
- 必要时删除异常主键记录。

### 13.4 后台没有数据显示

检查：

- `watermelon_data` 是否有数据。
- `field_id` 是否匹配设备编号前 4 位。
- 前端是否已登录。
- `/api/admin/watermelon/list?field_id=1001` 是否返回数据。
- 设备是否超过 30 分钟未上传，被前端标记为离线。

### 13.5 糖度异常偏高或偏低

检查：

- 是否测到黑色条纹区域。
- 是否多个通道饱和到 `51000`。
- 光源是否过近或过强。
- 传感器距离是否变化。
- 当前数据是否属于模型训练覆盖范围。
- 设备端和服务端糖度公式是否一致。

### 13.6 工业模式无法进入

检查：

- RTC 是否有效。
- WiFi 是否连上。
- `/api/device/time` 是否可访问。
- 设备是否拿到了合法 Unix 时间戳。

### 13.7 Web 登录后刷新又失效

当前 Session 是内存态，服务重启后所有登录状态都会失效，需要重新登录。

## 14. 安全与工程化注意事项

当前实现能满足原型和毕业设计演示，但仍有生产化改进空间：

- 数据库账号、密码和库名应迁移到配置文件或环境变量。
- 管理员默认密码初始化逻辑不应每次启动都执行。
- 设备 Token 应使用更高熵随机值。
- `/api/device/upload` 建议使用标准 HTTP 状态码。
- 管理接口应统一做 Session 校验，目前部分接口校验不完整。
- SQL 构造仍是字符串拼接，虽有转义，但长期建议迁移到预处理语句。
- `Session` 当前是内存态，服务重启后失效；正式系统可迁移到 Redis 或数据库。
- 单 MySQL 连接加互斥锁适合低并发；多设备高频上传时应使用连接池。
- `readFile("../../FieldMonitoringPlatform/...")` 依赖运行目录，systemd 必须设置正确 `WorkingDirectory`。
