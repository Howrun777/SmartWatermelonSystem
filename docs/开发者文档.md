# SmartWatermelonSystem 开发者文档

本文档面向项目后续维护、调试、部署和毕业设计答辩说明。内容以当前仓库实现为准，覆盖设备端、服务端、Web 前端、数据库、接口、离线补发、光谱数据采集和常见运维操作。

---

## 1. 项目定位

SmartWatermelonSystem 是一个基于 ESP32、多光谱传感器和 C++ 后端的西瓜无损糖度检测系统。

系统目标：

- 设备端采集 AS7341 多光谱数据、环境数据，并在本地显示糖度结果。
- 设备端支持断网本地缓存，联网后自动补发。
- 服务端负责设备认证、数据校验、糖度和成熟度计算、MySQL 入库。
- Web 端展示瓜田、西瓜糖度历史、环境历史和设备掉线状态。
- 项目支持真实 Brix 标注数据导出，用于后续模型校准。

## 3. 运行架构

```text
ESP32 设备端
  ├─ 采集 AS7341 光谱
  ├─ 采集 SHT31/BH1750 环境数据，传感器缺失时用 99 作为哨兵值
  ├─ 本地计算糖度并显示

  ├─ 每 6 秒刷新数据
  └─ 根据 collected_at 判断 30 分钟掉线
```

## 4. 设备端说明

### 4.1 配置入口

设备端主要配置在：

```text
FieldAcquisitionTerminal/config.h
```

关键配置：

```cpp
#define DEVICE_ID "1001-01-01"
#define WATERMELON_VARIETY "Kirin Watermelon"
#define DEVICE_TOKEN "device-token-001"
#define SERVER_URL "http://服务器地址:8080/api/device/upload"

#define I2C_SDA_PIN 27
#define I2C_SCL_PIN 22
#define BUZZER_PIN 26

#define ENV_UPDATE_INTERVAL 2000
#define AUTO_UPLOAD_INTERVAL 300000
```

注意：

- `DEVICE_ID` 格式为 `瓜田号-组号-编号`，服务端通过前 4 位提取 `field_id`。
- `DEVICE_TOKEN` 必须存在于 `device_auth` 表且 `status=1`。
- 云服务器上不要执行 PlatformIO 编译；设备端固件应在本地开发机或接入设备的环境中构建和烧录。

### 4.2 设备启动流程

```text
setup()
  ├─ 初始化蜂鸣器、传感器、LittleFS
  ├─ 初始化 TFT + LVGL + 触摸屏
  ├─ 尝试读取 DS3231 RTC
  ├─ RTC 无合法时间时尝试 WiFi 授时
  └─ 时间合法后解锁工业模式
```

时间合法判据：

```cpp
timestamp > 1600000000
```

### 4.3 工业模式测量流程

```text
onMeasureBtnClicked()
  ├─ 读取温湿度和光照
  ├─ 读取 AS7341 光谱
  ├─ 本地 SugarCalc 计算糖度
  ├─ 组装 JSON，带 collected_at
  ├─ 上传到 /api/device/upload
  ├─ 无论上传成功与否都写入 LittleFS
  └─ 更新工业模式 UI 历史表
```

工业模式顶部 UI：

- 左侧显示当前时间或测量状态。
- 右侧显示 WiFi 状态。
- WiFi 已连接：绿色 `WIFI-OK`。
- WiFi 未连接：红色 `WIFI-X`。
- 两者在同一行，使用 12 号字体，空间不足时状态文本会省略。

### 4.4 离线缓存与补发

本地缓存文件：

```text
/records.jsonl
```

单行 JSON 结构：

```json
{
  "ts": 1779031957,
  "brix": 6.8,
  "t": 25.1,
  "h": 60.2,
  "l": 1200,
  "up": 0,
  "spec": {
    "ch415": 12750,
    "ch445": 35540
  }
}
```

补发机制：

- WiFi 从断开变为连接时，启动补发。
- WiFi 一直在线时，每 15 秒检查是否有 `up=0` 的遗留记录。
- 补发前随机延迟 `0~10s`，避免多设备同时冲击服务器。
- 补发成功后调用 `markAsUploaded(ts)`。

已修复的关键问题：

- 秒级时间戳可能重复，旧逻辑会导致同一个 `ts` 的第二条记录一直 `up=0`，联网后反复上传。
- 当前逻辑在保存未上传记录时检测重复 `ts`，冲突时自动 `+1s`。
- `markAsUploaded()` 只标记 `ts` 匹配且 `up=0` 的第一条记录。
- `NetworkManager::uploadData()` 同时检查 HTTP 状态和 JSON `code == 200`，不再把业务失败误判为成功。

### 4.5 光谱采样注意事项

当前糖度模型使用：

```text
abs_λ = log10(chClear / (chλ + 1))
raw_brix = -1.2*abs_555 + 3.5*abs_640 + 6.8*abs_680 + 12.5*abs_NIR - 2.5
```

这是一版可运行的启发式 MLR 模型，不是最终标定模型。

真实采样时需要注意：

- 西瓜黑色条纹会显著改变反射率。
- 测条纹区域时，糖度结果容易升高。
- 测非条纹区域时，糖度可能大幅下降甚至被截断为 0。
- 多个通道达到 `51000` 时说明存在饱和，数据质量会下降。

建议采样规范：

- 同一批标定数据必须固定测点类型：条纹、非条纹不要混用。
- 每个西瓜至少测 5 次，取中位数或均值。
- 后续模型应使用归一化特征，例如 `chλ / chClear`、`ch680 / chNIR`，降低表皮颜色影响。
- 标定数据应包含多个真实 Brix 标签，单个 6.8Brix 样本只能用于观察稳定性，不能拟合完整糖度模型。

## 6. HTTP 接口

### 6.1 设备端接口

#### GET /api/device/time

用于设备开机或 WiFi 恢复后的授时。

响应：

```json
{
  "code": 200,
  "msg": "时间同步成功",
  "data": {
    "server_time": 1779031957
  }
}
```

#### POST /api/device/upload

请求头：

```text
device_id: 1001-01-01
token: device-token-001
Content-Type: application/json
```

请求体：

```json
{
  "collected_at": 1779031957,
  "temperature": 25.2,
  "humidity": 62.3,
  "light": 1200,
  "spectrum_json": {
    "ch415": 12750,
    "ch445": 35540,
    "ch480": 33549,
    "ch515": 51000,
    "ch555": 51000,
    "ch595": 51000,
    "ch640": 51000,
    "ch680": 51000,
    "chClear": 51000,
    "chNIR": 29668
  }
}
```

成功响应：

```json
{
  "code": 200,
  "msg": "数据上传成功",
  "data": {
    "sugar_brix": 6.8,
    "maturity_score": 0.544,
    "server_time": 1779031960
  }
}
```

失败响应仍可能使用 HTTP 200 返回，因此设备端必须检查 JSON `code`。

常见业务失败：

- `401`：设备认证失败、Token 错误或设备禁用。
- `400`：JSON 格式错误、瓜田未注册。
- `500`：数据库写入失败，常见于重复时间戳主键冲突。

## 10. 关键风险与后续优化

### 10.1 光谱模型风险

当前糖度模型仍是初版经验公式，受以下因素影响明显：

- 西瓜黑色条纹和非条纹表皮反射差异。
- 传感器距离和角度。
- 光源是否完全开启。
- 通道饱和，尤其是多个通道达到 `51000`。
- 数据集真实 Brix 标签不足。

建议后续优化：

- 加入饱和检测和低亮度检测。
- 使用多点采样中位数。
- 用真实 Brix 数据重训模型。
- 使用归一化特征减少表皮颜色影响。
- 可将条纹/非条纹作为两类场景分别建模。

### 10.2 服务端工程风险

当前服务端可运行，但仍有工程化改进空间：

- 数据库连接参数应从代码迁出。
- `/api/device/upload` 对业务失败仍使用 HTTP 200，设备端已兼容，但后续可改成真实 HTTP 状态码。
- `INSERT` 主键冲突目前返回失败，后续可考虑 `INSERT ... ON DUPLICATE KEY UPDATE` 或服务端自动偏移时间戳。
- `MySQLDriver` 是单连接加互斥锁，当前规模可用，高并发时应使用连接池。

### 10.3 设备端风险

- LittleFS 容量超过 80% 时会删除最早一半记录。
- RTC 无效且 WiFi/服务端不可用时，工业模式不会解锁。
- WiFi 重连是非阻塞的，首次授时仍可能等待。
- 本地和服务端糖度公式需要保持一致，否则 UI 和后台结果会出现差异。

## 11. 快速排障

### 设备无法进入工业模式

检查：

- DS3231 是否有合法时间。
- WiFi 是否能连接。
- `/api/device/time` 是否能返回 `server_time`。
- 设备屏幕状态是否显示 `RTC/WIFI/SVR` 错误组合。

### 设备一直显示上传或反复补发

检查：

- `watermelon_data` 是否存在重复主键冲突。
- 设备端 `StorageManager::markAsUploaded()` 是否为最新逻辑。
- `NetworkManager::uploadData()` 是否检查 JSON `code == 200`。

### 后台没有数据

检查：

- `smartwatermelon.service` 是否 active。
- 8080 端口是否监听。
- `device_auth` 中设备是否启用。
- `field_production` 中是否存在设备对应 `field_id`。
- `watermelon_data` 是否有记录。

### 光谱糖度明显异常

检查：

- 是否未开灯采样。
- 是否采到黑色条纹，且训练数据主要来自非条纹。
- 是否多个通道达到 `51000` 饱和。
- 是否传感器距离变化太大。
